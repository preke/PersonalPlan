1. 对论文现有评估的诊断

当前 5-Experiments.tex 中的评估配置是：

  • 5 项规则式结构指标：SV、AR、DC、ATR、GED-sim。
  • 3 项 LLM judge：Personalization (Pers.)、Feasibility (Feas.)、Satisfaction (Sati.)。
  • 三层协议：Tier 1 静态、Tier 2 在线 MAS（仅作 Plan Executability Rate）、Tier 3 对话级 rubric。
维度评估问题 / 缺口
结构性 合理 SV/AR/DC/ATR/GED-sim 是规则式的、廉价、与多数 ACL/EMNLP plan-quality 论文一致（参考 PLAN-TUNING、WebAgent-R1）。可保留。
个性化 薄 只有 Pers. 一项 LLM judge。缺一个反事实信号：同 query 不同 profile，plan 是否真的变？若不变就说明 PAD 没起作用——这是 RQ2 / RQ3 的核心。需要补 Profile-Variance Score 与 Profile-No-Profile Gap。
教学法 缺 Feas. 关注 dependency / 工具，不评估教学结构（阶段覆盖、阶梯难度、苏格拉底比例）。论文方法明明引入了 phase coverage + subtask precedence 两项 reward（§3.3 R_ped），却没有对应的 evaluation 指标——RQ3 没法支撑。需补 Pedagogy Score（仿 KELE 4 指标）。
执行可行性 单点 只有 Executability Rate（端到端跑通比例）。实际上你们已经实现了完整的 4 项 LLM 检查（Instruction Fidelity / Workflow Completeness / Interaction Quality / Content Correctness）——白白没用。这是 Tier 2 最强的判别器，应该单独成主评估。
编程专属 缺 MAP-PPL 全是编程查询，但评估里没有任何代码相关指标。SafeCodeInterpreterTool 已经能跑 14 种语言、能拿到真实 stdout/stderr——这些信号没进评估。需补 Code Execution Rate、Code Correctness Rate、Learner Solve Transition。
对话级评估 声明了未落地 论文里 Tier 3 写了"dialogue-level rubric"，但缺具体 rubric。现有 evaluator 的 Check 3（Interaction Quality）+ Check 4（Guidance Effectiveness）正好就是 dialogue-level rubric 的落地，可直接接入。
判卷者可信度 声明了未落地 论文写了"κ ≥ 0.65 / ICC ≥ 0.7"——但没有具体的 calibration 子集大小、anchor 校准方式、judge family 配对。需要明确 100 条 pilot + Aligning-Pedagogy 风格的 M-judge 乘积。
核心修订点：保留 5 项规则式结构指标；保留 LLM-judge Pers./Feas./Sati. 但细化 rubric；新增 Pedagogy Score、Profile-Variance / Profile-No-Profile、4 项编程专属指标、4 项 Tier 2 LLM 检查。再加上跨模型族 judge + 人工 κ/ICC 校准。最终 8 项 → 18 项（分三层）。

2. 小同行评估方法综述（ACL/EMNLP 2024–2025）

从 related_papers/ 下 20 篇 ACL/EMNLP 论文中，按"评估侧重点"归类：

2.1 静态 plan 质量类

论文评估手段启示
PLAN-TUNING (EMNLP 2025) 双指标 reward——R_plan（Gemini-2.0-Flash 与 gold plan 相似度，∈ [0,1]）+ R_ans（验证答案正确性，∈ {0,2}）；用 PlanGEN verifier 打 ≥80 分作为 SFT 训练数据过滤。 Plan-quality 用 LLM-judge 相似度，answer correctness 用规则式——和我们 GED-sim + 代码可执行性的组合完全对应。
MPO (Findings EMNLP 2025) SFT 之后用 GPT-4o 对每个 meta-plan 评 3 个维度（correctness / followability / standardisation）+ MC rollout completion rate。 训练时用 rollout 信号（不调 LLM judge），评估时再用 GPT-4o 复核——和我们"GRPO 训练全规则式、Test 才调 judge"一致。
iTool (EMNLP 2025) BFCL v3 五个子集（Non-live / Live / Multi-turn / Rel / Irrel）+ API-Bank L1/L2，每个子集用精确字段比对（参数名 / 参数值 EM）。 把"plan 结构错误"细分成"参数名错"vs"参数值错"——启示我们 SV 应该分顶层 key 缺失 vs 类型错误 vs schema 嵌套错误。

2.2 多智能体 / 执行类

论文评估手段启示
AgentGym (ACL 2025) 11 个 env 端到端跑通 + reward；OOD 测试用 ALF/Baby-OOD、AM、ST 三个 held-out env。 "plan 喂进 env 跑通"是最强 discriminator——我们已经有 CrewAI runtime，应该把执行评估提到主表。
WebAgent-R1 (EMNLP 2025) WebArena-Lite 165 个验证 task，规则式二元 reward（URL match / string match / program exec）；OOD 在 5 个 WebVoyager 域上测。 Tier 2 的执行检查应该用规则式判定（代码跑没跑通、输出对没对），不用 LLM。LLM 只判语义合理性。
AgentPro (EMNLP 2025) MCTS 每步 m=3 simulated completions，PRM 标"+/-"；trajectory reward 是 step 平均。 启示：plan 的"过程级"评估应该用 MCTS rollout 而不是单次 run——但我们没必要做 MCTS，1 次 live run 即可，因为 plan 是静态产物。
LRPLAN (Findings EMNLP 2025) TravelPlanner 用 commonsense pass-rate / hard pass-rate / final pass-rate；TimeArena-Static 用 FCR/TCR/Avg-Progress/Dependency Violations。 把"硬约束 pass-rate"和"软约束 pass-rate"分开报——直接对应我们 R_hard vs R_soft 的设计；可以加 "Hard Constraint Violation Rate" 主表列。

2.3 教学法 / 个性化类（最重要）

论文评估手段启示
KELE (Findings EMNLP 2025) 4 项 single-turn 结构指标：PRR (Pedagogical Rule Reasonableness)、NDAR (No-Direct-Answer Rate)、SPR (Stage Progression Rate)、IAR (Inductive Aptness Rate)；100 条人工 κ 0.65–0.75；5 项 multi-turn GPT-4o 1–5 评分 + 研究生 ICC 0.68–0.83；ROUGE-1/2/L + BLEU-4。 直接搬过来作我们 Pedagogy Score 的 4 个子维度，但把"5 苏格拉底阶段"换成我们的 6 个 instructional phase（probe / retrieve-demonstrate / apply / validate / feedback / consolidate）。κ/ICC 阈值照搬。
Aligning-Pedagogy (EMNLP 2025) 双指标 reward——r_sol（K=8 个 student 采样的 post-dialog solve rate，对 gold 验证）+ r_ped（M 个独立 LLM-judge 的 乘积，含 Answer-Leakage + Helpfulness）；λ 参数 sweep 出 Pareto 曲线；训练 judge 用 Qwen2.5-14B，评估 judge 用 Gemma3-27B（不同模型族）。 这是最值得抄的设计：(1) M-judge 乘积比平均更敏感；(2) 训练 / 评估 judge 模型族必须分开；(3) λ Pareto 图本身就是 RQ4 的图。我们 GRPO reward 权重 sweep 5 个 run 就是为这张图。
TreeInstruct (Findings EMNLP 2024) Socratic Debugging Benchmark + MULTI-DEBUG（自建 50 LeetCode 多 bug 集）；定量指标 = 与 ground-truth fix 同构的 success rate；定性 = Relevance / Indirectness / Logical Flow / 平均轮数。 "plan 同构性"作 ground-truth 对比——和我们 GED-sim 一致。"Indirectness"（不直接给答案）应该作为 Pedagogy Score 的一个子项（NDAR）。
Personality-aware Student Sim (EMNLP 2024) 500 dialogue × 7 scaffolding 类型 heatmap；BFI 44 题 Pearson 相关；500 dialogue 中 50 条人工 validation。 启示：profile 评估应该看"plan 在不同 profile 下的 scaffolding 类型分布"——但是我们短期内不做 scaffolding 分类（成本太高），保留 7 类作 case study。
Student Data Paradox (Findings EMNLP 2024) TruthfulQA / ARC / MMLU / WinoGrande 等 9 个 benchmark 测 tutor 模型在通用能力上的"side-effect"。 启示：profile-conditioned tutor 不能伤通用能力——但我们只训 LoRA + GRPO 1 ep，理论上影响很小；可以可选加一个"backbone 通用能力保持"sanity check（MMLU + HumanEval 各 50 条）。

4. Tier 1 — 静态 Plan 质量（11 项）

4.1 6 项规则式结构性指标

SV — Schema Validity

定义：\(\mathrm{SV}(\mathcal{P}) = \mathbb{I}[\mathrm{validate}_{\mathrm{schema}}(\mathcal{P})] \in \{0,1\}\)。\(\mathrm{validate}_{\mathrm{schema}}(\mathcal{P})\) 取真当且仅当下列 4 个条件全部成立：

  1. \(\mathcal{P}\) 可被解析为合法 JSON；
  2. 顶层 4 个必需 key 齐全：agents / subtasks / steps / execution_order；
  3. 所有字段类型与声明一致（如 agents 是 array of AgentSpec, execution_order 是 array of step_id string）；
  4. 所有嵌套对象（AgentSpec / SubtaskSpec / StepSpec / LoopSpec）均通过 Pydantic 校验。

实现：直接复用 plan_mapper_fixed/models.py 里的 PlanPayload.model_validate()。完整公式见 §11.1。

来源：PLAN-TUNING 的"两阶段过滤"第一步。

AR — Acyclicity Rate

定义：plan 的 step 依赖图 G=(S, O)（O 是 depends_on 边）通过拓扑排序的比例。1/0。

实现：networkx.is_directed_acyclic_graph(G)。

来源：LRPLAN 的 Dependency Violations。

DC — Dependency Completeness

定义：\(\mathrm{DC}(\mathcal{P}) = |\mathcal{D}_{\mathrm{sat}}| / |\mathcal{D}_{\mathrm{tot}}|\)，其中 \(\mathcal{D}_{\mathrm{sat}}\) 是 depends_on 指向的 step 实际存在于 plan 内的依赖数，\(\mathcal{D}_{\mathrm{tot}}\) 是所有声明依赖数。∈ [0,1]。完整公式见 §11.1。

实现：遍历 step.depends_on + step_index 查找。

来源：论文 §3 已定义，沿用。

ATR — Agent-Tool Relevance

定义：\(\mathrm{ATR}(\mathcal{P}) = \frac{1}{N}\sum_{j=1}^{N} (I^{(j)}_{A\to S} + I^{(j)}_{T\to A} + I^{(j)}_{T\to S}) / 3 \in [0,1]\)。对每个 step \(s_j\)，三个 0/1 indicator 的判定条件为：

  1. \(I^{(j)}_{A\to S} = 1\) 当且仅当 agent 角色关键词出现在 subtask 的推荐角色表（rubric YAML）内，或 agent role 与 subtask 描述的 embedding cosine \(> 0.30\)；
  2. \(I^{(j)}_{T\to A} = 1\) 当且仅当 \(s_j.\mathrm{tool} \ne \varnothing\) 且 \(s_j.\mathrm{tool} \in \mathrm{ag}(s_j).\mathrm{tools}\)（若 tool 为 null 则直接置 0）；
  3. \(I^{(j)}_{T\to S} = 1\) 当且仅当 \(s_j.\mathrm{tool}\) 出现在 subtask 所属类别的推荐工具集中（同样的 mapping 表，如 SecurityAdvisor → {nmap, burp, ...}）。

实现：论文 §3 已定义。注意 当 step.tool=null 时 \(I_{T\to A}\) 和 \(I_{T\to S}\) 直接置 0，避免无工具步骤被错误扣分。完整公式见 §11.1。

TBV — Tool-Budget Validity 新增

定义：\(\mathrm{TBV}(\mathcal{P}) = \frac{1}{|\mathcal{S}_{\mathrm{tool}}|}\sum_{s \in \mathcal{S}_{\mathrm{tool}}} v(s) \in [0,1]\)（若 \(|\mathcal{S}_{\mathrm{tool}}|=0\) 则取 1）。对每个带 tool 的 step \(s\)，\(v(s) = 1\) 当且仅当下列 2 个条件同时成立：

  1. \(s.\mathrm{tool}\) 来自预定义 8 工具池 \(\Pi\)：{FirecrawlSearchTool, RagTool, CodeInterpreterTool, DirectoryReadTool, FileReadTool, FileWriterTool, CodeDocsSearchTool, ArxivPaperTool}；
  2. \(s.\mathrm{tool}\) 出现在该 step 所属 agent 的 tools 列表内（\(s.\mathrm{tool} \in \mathrm{ag}(s).\mathrm{tools}\)）。

实现：直接复用 compiler.py:compile_plan() 的 tool_binding_errors。完整公式见 §11.1。

来源：iTool 提出"77% 的 tool 错误集中在 Parameter-Value / Name"——我们等价地拆出 tool-binding 错误率。已经在 compiler 里实现，不计入额外开发成本。

GED-sim — Structural Similarity to Gold

定义：\(\mathrm{GED\text{-}sim}(\mathcal{P}, \mathcal{P}^{\star}) = 1 - \mathrm{GED}(G', G^{\star}) / \max(|\mathcal{S}'|+|\mathcal{O}'|, |\mathcal{S}^{\star}|+|\mathcal{O}^{\star}|)\)，其中 \(\mathrm{GED}\) 是 graph edit distance，\(G'\) 是 candidate 的 (step, dependency) 图，\(G^{\star}\) 是 gold plan 的同结构图。∈ [0,1]。完整公式见 §11.1。

实现：networkx.graph_edit_distance(G1, G2, timeout=2.0)，timeout 2s 防止大图卡死。

来源：论文 §3 已定义 + TreeInstruct 的"plan 同构性"思想。

4.2 4 项 LLM-judge plan-quality 指标

Pers. — Personalization（3 个子维度）

3 个子维度（独立 1–5 Likert，5/3/1 anchored）：

  1. Skill Match：plan 起点是否匹配 learner 的实际技能水平（不解释已知 / 不跳过缺口）
  2. Goal Orientation：每个 subtask 是否都服务于 query 的核心问题
  3. Background Adaptation：例子、agent role、解释方式是否引用 about_me 与 top_tags

聚合：3 子维度均值，归一到 [0,1] via (mean − 1) / 4。

实现：现有 prompt 模板基础上拆 3 次独立 LLM call（防止 halo effect）。

来源：KELE 5/3/1 anchored + Personality-Sim 子维度独立打分。

Feas. — Feasibility（3 个子维度）

  1. Dependency Ordering：execution_order 与 depends_on 一致、无环
  2. Task Decomposition：subtask / step 粒度合理（不太粗也不太细）
  3. Agent-Tool Appropriateness：agent 角色与 step 匹配、tool 调用合理

聚合 + 实现：同 Pers.

判卷要点：把 gold plan 的 execution_order 作为 reference 给 judge（reference-grounded），让 judge 判 candidate "是否同样合理"——参考 Aligning-Pedagogy 的做法，比 zero-shot 判定准很多。

Ped. — Pedagogy 新增（4 个子维度，仿 KELE）

这是论文里 R_ped reward 在评估上的对应，目前完全缺失。4 个子维度：

  1. PRR (Pedagogical Rule Reasonableness)：subtask 之间的先后关系是否符合学习规律（先概念后实现、先实现后调试 …）。LLM 5/3/1 anchored。
  2. NDAR (No-Direct-Answer Rate)：plan 是否避免在前期就直接给 accepted_answer。LLM 评是否包含"先给结论后让学生套用"反模式。1/0 比例（NDAR 高代表"不直接给答案"）。
  3. SPR (Stage Progression Rate)：6 个标准 instructional phase（probe / retrieve-demonstrate / apply / validate / feedback / consolidate）的覆盖率。\(\mathrm{SPR}(\mathcal{P}) = \frac{1}{|\Phi|}\sum_{\phi \in \Phi} \mathbb{I}[\exists s \in \mathcal{S}: \mathrm{phase}(s) = \phi]\)。规则式（用 phase keyword + LLM 二分类辅助）。
  4. IAR (Inductive Aptness Rate)：plan 是否采用 inductive style（从具体例子归纳到一般规律）vs deductive style（先抛出规则）。LLM 5/3/1。inductive 更适合初学者 profile，deductive 更适合高级 profile——根据 profile 加权。

聚合：Ped. = (PRR + NDAR + SPR + IAR) / 4，∈ [0,1]。

来源：KELE 直接 port 4 项；Aligning-Pedagogy 的 r_ped 多 judge 乘积思想用在 PRR/IAR 上。

实现成本：3 个 LLM call（PRR/NDAR/IAR）+ 1 个规则式（SPR）。每条 plan ~$0.03（GPT-5）。

5. Tier 2 — 多智能体执行评估（4 项 = 1 执行可行 + 3 教学效果）

Tier 2 直接回答论文最 high-stakes 的两个判断：
  1. (Q1) Plan 是否真的能被 CrewAI 正确执行？——把 plan JSON 喂进 PlanRuntime，跑 Teacher×Student dialogue，看声明用 CodeInterpreterTool 的 step 是否在 execution log 里有真实 tool 调用事件、sandbox 是否成功执行（exit code = 0、无 unhandled exception）、所有 step 是否按 execution_order 跑完。这是不可 hack 的 objective signal——代码跑不通就是跑不通。

5.2 Axis A — Plan 执行可行性（1 项 headline）

EVR — Execution Validity Rate headline · Q1 v1.1 revised

判断：plan 是否真的端到端跑通 + 所有声明 CodeInterpreterTool 的 step 在 execution log 里有真实 tool 调用事件 + sandbox 成功执行（exit_code = 0、无 unhandled exception）。

定义（5 个 binary sub-check 的乘积，等价于 KELE Workflow Completeness 3 子检查 + WebAgent-R1 program-exec binary reward 在含 code execution 的设定下的扩展）：

\[ \mathrm{EVR}_r \;=\; \underbrace{\mathbb{I}[\mathrm{cov}_r] \cdot \mathbb{I}[\mathrm{loop}_r] \cdot \mathbb{I}[\mathrm{flow}_r]}_{\text{WC.a,b,c：流程完整}} \,\cdot\, \underbrace{\mathbb{I}[\mathrm{CER}_r = 1]}_{\text{所有 code step 真调 tool}} \,\cdot\, \underbrace{\mathbb{I}[\mathrm{CCR}_r \ge \tau]}_{\text{sandbox 执行成功，}\tau=1} \]

其中 cov / loop / flow 是 KELE WC 子检查；CER 是 tool-call invocation rate（execution log 中存在 CodeInterpreterTool 调用事件，纳入式检测）；CCR 是 sandbox 执行成功率（exit_code = 0 且无 unhandled exception，对标 WebAgent-R1 program-exec binary reward）。报告 \(\mathrm{EVR} = \mathbb{E}_r[\mathrm{EVR}_r]\)。这是 Tier 2 的 Q1 headline。完整公式与 v1.1 修订说明见 §11.4。

来源：AgentGym task success rate \citep{xi2025agentgym}；WebAgent-R1 program-exec binary reward \citep{wei2025webagentr1}；LRPLAN final pass-rate (commonsense ∧ hard) \citep{karthikeyan2025lrplan}；HumanEval pass@1 用于 code-step CCR \citep{chen2021codex}。论文里推荐写"Following \citet{xi2025agentgym,wei2025webagentr1}, we report task success rate (EVR)..."。

5.3 Axis B — MAS 教学效果（3 项 headline，每项对应 Q2 的一个子问题）

这 3 项 都直接 port 自 prior art——PAS = KELE PRR、PQS = mean(KELE NDAR + SPR + IAR)、rsol = Aligning-Pedagogy。reviewer 一看就知道"这是公认指标 + cite"，可信度高。

PAS — Personalization Alignment Score headline · Q2.a 个性化

判断：MAS 教学过程中，教师 agent 实际说的话是否反映 learner_profile 的关键字段（age / knowledge_level / learning_style / native_language / interests）。

定义（直接复用 KELE PRR = Personalization Reflection Rate）：对每个 teacher_output 由 LLM judge 判定与 learner profile 的对齐档位 \(p_u \in \{2, 1, 0\}\)（well-aligned / partially / not），归一到 [0,1]：

\[ \mathrm{PAS}_r \;=\; \frac{1}{2|\mathcal{U}_r^{\mathrm{tch}}|}\sum_{u \in \mathcal{U}_r^{\mathrm{tch}}} p_u, \qquad \mathrm{PAS} \;=\; \mathbb{E}_r[\mathrm{PAS}_r] \in [0,1] \]

\(\mathcal{U}_r^{\mathrm{tch}}\) 是 run \(r\) 中所有 teacher_output（含 learner-facing step + loop iterations）。完整公式 + judge rubric 见 §11.5。

来源：KELE PRR \citep{peng2025kele} 直接 port；与 evaluator.py 现有 Check 1 (IF) 中 "符合 learner profile" 子条件协同（IF 在 appendix 作 sub-check 报告）。论文写"Following \citet{peng2025kele}, we measure the Personalization Reflection Rate (PRR)..."。

PQS — Pedagogical Quality Score headline · Q2.b 教学法

判断：MAS 教学是否遵循好教学法——(i) 不直接给答案（鼓励学生自己想）；(ii) 有 scaffolding phase（introduce → guide → consolidate）；(iii) 多用提问而非告知（Socratic）。

定义（直接采用 KELE 3 个 pedagogical sub-metrics 的算术均值）：

\[ \mathrm{PQS}_r \;=\; \frac{1}{3}\!\left(\, \mathrm{NDAR}_r + \mathrm{SPR}_r + \mathrm{IAR}_r \,\right), \qquad \mathrm{PQS} \;=\; \mathbb{E}_r[\mathrm{PQS}_r] \in [0,1] \]

其中：

  • NDAR (Non-Direct-Answer Rate)：teacher_output 中不直接 reveal accepted_answer 的比例（LLM judge 判 reveal / partial / none，归一到 [0,1]）；
  • SPR (Scaffolding Phase Rate)：plan execution 是否依序触发 introduce / guide / consolidate 三 phase（rule-based 按 step.type 或 LLM-tagged）；
  • IAR (Inquiry-to-Answer Ratio)：teacher utterance 中"提问"个数 / "告知"个数，clip 到 [0,1]（≥1.0 视为充分提问）。

完整公式 + 每子指标 rubric 见 §11.5。

来源：KELE NDAR/SPR/IAR \citep{peng2025kele} 3 项 metric 直接 port；与现有 evaluator.py Check 3 (IQ) + Check 4b (Guidance Trace) 协同——IQ / CCGE.Guidance 在 appendix 作 sub-check 验证。论文写"We adopt the three pedagogical sub-metrics (NDAR, SPR, IAR) from \citet{peng2025kele}, reported as their mean (PQS)..."。

rsol — Post-Dialog Solve Rate headline · Q2.c 学生学会

判断：plan 跑完后，学生 agent（固定 Llama-3.1-8B-Instruct）在最后一个 subtask 内独立交出的代码，是否与 accepted_answer 功能等价。这是 Tier 2 最 high-stakes 的指标——直接对应论文 abstract 里"学生真的被教会了"的论断。

定义（直接 port from Aligning-Pedagogy 的 \(r_{\text{sol}}\)）：

\[ r_{\mathrm{sol}} \;=\; \frac{1}{|\mathcal{R}_{\mathrm{valid}}|}\sum_{r \in \mathcal{R}_{\mathrm{valid}}} \mathbb{I}\!\left[\, \mathrm{code}_r^{\mathrm{stud,final}} \,\equiv_J\, \alpha_r \,\right] \]

其中 \(\mathcal{R}_{\mathrm{valid}}\) 排除 "perfect_from_start" 话题盲区失败 run（用 IQ check 标记），\(\alpha_r\) 是该 run 的 accepted answer，\(\equiv_J\) 是 LLM judge (GPT-5) 判定的功能等价。论文同时报 \(|\mathcal{R}_{\mathrm{valid}}|/|\mathcal{R}|\) 防止读者误解分母。完整公式见 §11.5。

来源：Aligning-Pedagogy \citep{dinucu2025aligningpedagogy} 的核心 reward \(r_{\text{sol}}\) 直接 port（math tutoring → programming tutoring）；与现有 evaluator.py Check 4c (Final Output Correctness) 协同。论文写"Following \citet{dinucu2025aligningpedagogy}, we report the post-dialog solve rate (\(r_{\text{sol}}\)): the fraction of runs where the simulated student produces code functionally equivalent to the Stack Overflow accepted answer..."。

6. Tier 3 — 学习者偏好（2 项）

Sati. — Profile-Conditioned Satisfaction

定义：profile-conditioned LLM judge 在 candidate plan vs gold plan 之间做 pairwise 选择，不告知 judge 哪个是 gold。Sati. = (candidate 被偏好的比例) + 0.5 × (Tie 比例)。

Judge prompt 用 4 criteria（按优先级）：

  1. Skill Match：起点是否适合 learner 已声明的技能
  2. Engagement & Learnability：是否更易于跟随
  3. Structural Appropriateness：结构是否更可执行
  4. Personal Relevance：是否真的个性化（非通用模板）

协议：

  • AB / BA 顺序各跑一次，取均值（消除 position bias）
  • 不一致（AB 选 A、BA 选 B）→ 计为 Tie
  • 主判 GPT-5 + 复核 Claude Opus 4.6 + Gemini 3 Pro（M-judge 乘积）

聚合：M-judge 乘积——\(\mathrm{Sati.} = \prod_{m=1}^{M} w_m\) 而非平均，参考 Aligning-Pedagogy。乘积比平均更敏感：一个 judge 不喜欢 → 整体掉很多。完整公式见 §11.2。

来源：Aligning-Pedagogy M-judge product；MAgIC paired t-test。

JCC — Judge Cross-Consistency 新增

定义：3 个 judge（GPT-5 / Claude Opus 4.6 / Gemini 3 Pro）在同一对 pairwise 上的 3-way agreement 比例（Krippendorff α）。

用途：报告在 appendix 作为 Sati. 的 可信度证明。α < 0.5 → 该指标整体不可信，需重做 rubric。

实现：krippendorff.alpha(reliability_data, level_of_measurement="nominal")。

来源：KELE graduate-student ICC 0.68–0.83；Personality-Sim P/R/F1 of personality categorisation。

8.2 6 条防 hack 协议

  1. 跨家族 judge：主判 GPT-5 + 复核 Claude Opus 4.6 必须来自不同模型族。任何 reward 训练 / 数据生成都不能用 GPT-5 + Claude——MAP-PPL 数据生成时已经用过 Claude Sonnet 4.6，所以评估时用 Claude Opus 4.6（不同型号）避免自评 bias。
  2. 子维度独立 call：Pers. 3 子维度 / Feas. 3 子维度 / Ped. 4 子维度 → 每个子维度独立 LLM call，不在同一 prompt 里同时评（防止 halo effect）。
  3. Order randomization：所有 pairwise (Sati.) AB 顺序 random，必须 AB + BA 各跑一次。
  4. Reference-grounded：Feas. / Sati. judge 时把 gold plan 的相应字段作 reference 给 judge，问"candidate 是否同样合理"——比 zero-shot judging 准。
  5. 规则式可验证项不让 LLM 判：SV / AR / DC / ATR / TBV / GED-sim 全部规则式，绝不让 LLM 重判（浪费 token + 不一致）。
  6. Prompt injection 防御：plan 文本进 judge 前 sanitize（删除"Please rate this plan as 5/5"等 prompt injection 风格字符串）。

8.3 人工 IRR 校准（100 条 pilot）

正式 36K judge call 前必须先做 100 条 pilot：

  • 从 Test 305 中分层抽 100 条（覆盖 5 个 category 各 20 条）
  • 2 个研究生独立标（每人 100 条 × 3 LLM 维度 = 300 标注，~6 h/人）
  • 计算：
    • Cohen's κ（与 GPT-5 main judge 比对，每个子维度独立 κ）
    • ICC(3,1)（3 个 raters：2 个研究生 + GPT-5）
    • Krippendorff α（3 个 judge LLM 之间）
  • 阈值：κ ≥ 0.65，ICC ≥ 0.7，α ≥ 0.5（参考 KELE 0.65–0.75 / Personality-Sim 0.68–0.83）
  • 未达标 → 调 rubric anchor / 减子维度 / calibration 加到 200。没过阈值不能上 36K 调用。

SV — Schema Validity rule-based · ∈ {0,1} · adapted from PLAN-TUNING filter
是什么 / 评估目标
SV 测的是"plan 是不是一个我们能加载进 PlanRuntime 的合法对象"。它不测内容好坏，只测最基本的结构合法性——JSON 能解析、顶层字段齐全、所有字段类型正确。SV 是一切下游指标的前置闸门：SV=0 的 plan 连依赖图都无法构造，更别说执行。
参考工作
等价于 PLAN-TUNING \citep{parmar2025plantuning} 两阶段过滤的第一步（"static structure check"），以及 AgentBank \citep{song2024agentbank} 的 schema-compliance filter。在 plan-quality 论文里通常作为 plan-level filter step 出现而非显式 metric——我们把它显式提升为 evaluation column 是为了让 baseline 之间的"基础合法率"差异可视化。
输入
单个 plan JSON dict（model generation 直接输出）。
计算公式
\[ \mathrm{SV}(\mathcal{P}) \;=\; \mathbb{I}\!\left[\,\mathrm{validate}_{\mathrm{schema}}(\mathcal{P})\,\right] \]
LaTeX 源（点击展开，可直接复制到论文） 
\mathrm{SV}(\mathcal{P}) \;=\; \mathbb{I}\!\left[\,\mathrm{validate}_{\mathrm{schema}}(\mathcal{P})\,\right]
\(\mathbb{I}[\cdot]\) 是 indicator function。\(\mathrm{validate}_{\mathrm{schema}}(\mathcal{P})\) 取真当且仅当下列 4 个条件全部成立：
  1. \(\mathcal{P}\) 可被解析为合法 JSON；
  2. 顶层 4 个必需 key 齐全：agents / subtasks / steps / execution_order；
  3. 所有字段类型与声明一致（如 agents 是 array of AgentSpec, execution_order 是 array of step_id string）；
  4. 所有嵌套对象（AgentSpec / SubtaskSpec / StepSpec / LoopSpec）均通过 Pydantic 校验。
LaTeX 源（含条件枚举的论文写法） 
\mathrm{SV}(\mathcal{P}) \;=\; \mathbb{I}\!\left[\,\mathrm{validate}_{\mathrm{schema}}(\mathcal{P})\,\right],

\text{where } \mathrm{validate}_{\mathrm{schema}}(\mathcal{P}) \text{ holds iff}
\text{(i) } \mathcal{P} \text{ parses as JSON};
\text{(ii) top-level keys} = \{\texttt{agents}, \texttt{subtasks}, \texttt{steps}, \texttt{execution\_order}\};
\text{(iii) every field matches its declared type};
\text{(iv) all nested specs pass the Pydantic schema.}

AR — DAG Validity (Acyclicity Rate) rule-based · ∈ {0,1} · inverse of LRPLAN Dep-Violations
是什么 / 评估目标
AR 测的是 plan 的 step 依赖图能不能被拓扑排序——也就是 PlanRuntime 能不能找到一条合法的执行顺序。任何环（自环 / 互相依赖 / 长环）都让 runtime 卡在循环等待。AR 不管 dep ID 是否真的存在（那是 DC 的事），只管图本身有没有环。
参考工作
等价于 LRPLAN \citep{karthikeyan2025lrplan} TimeArena-Static 报告的 "Dependency Violations" 指标的取反。判定算法（Kahn 拓扑排序）是 CS 通用工具，NetworkX is_directed_acyclic_graph 直接调用。Plan-quality 类论文几乎都报某种形式的"依赖合法性"，名字略有差异但语义一致。
输入
已通过 SV 的 plan（否则不进入 AR 计算，直接记 AR=0）。需提取 step list 和 depends_on 列表构造依赖图。
计算公式
构造有向图 \(G = (\mathcal{S}, \mathcal{O})\)，其中 \(\mathcal{S}\) 是所有 step.id，\(\mathcal{O} = \{(u, v) \mid u \in v.\mathrm{depends\_on}\}\)。
\[ \mathrm{AR}(\mathcal{P}) \;=\; \mathbb{I}\!\left[\,G\ \text{is a DAG}\,\right] \]
LaTeX 源 
\mathrm{AR}(\mathcal{P}) \;=\; \mathbb{I}\!\left[\,G\ \text{is a DAG}\,\right]
参考代码
import networkx as nx

def ar(plan_json: dict) -> int:
    G = nx.DiGraph()
    for st in plan_json["output"]["subtasks"]:
        for step in st["steps"]:
            G.add_node(step["id"])
            for dep in step.get("depends_on", []):
                G.add_edge(dep, step["id"])
    return int(nx.is_directed_acyclic_graph(G))
✅ 通过示例（AR = 1）
4 种典型合法形状：
  • 线性链：S1-1 → S1-2 → S2-1，可拓扑排序 [S1-1, S1-2, S2-1]
  • 钻石分支：S1-1 → {S1-2, S1-3} → S2-1（S2-1 同时依赖 S1-2 和 S1-3）
  • 跨 subtask 串联：S1 的最后 step 是 S2 的前置；S2 的最后 step 是 S3 的前置
  • 无依赖的并列 step：所有 step.depends_on=[]，空图 trivially 是 DAG
❌ 不通过示例（AR = 0）
4 种典型环结构：
  • 自环：{"id": "S1-1", "depends_on": ["S1-1"]}——弱 baseline ~3-5% plan 会犯
  • 2-step 环：S1-1.depends_on=["S1-2"] 且 S1-2.depends_on=["S1-1"]（LLM 把"先 A 后 B"和"先 B 回顾 A"双向连了）
  • 长环：S1-1 → S1-2 → S2-1 → S1-1，3+ step 形成循环
  • 跨 subtask 隐环：表面上 plan 是按 S1→S2→S3 写的，但某个 step depends_on 反向指回去——这种肉眼难发现，但 is_directed_acyclic_graph 一查就出
方向 / 阈值
↑ higher better。R_hard 闸门第 2 条。SOTA baseline ~0.85-0.93，弱 baseline ~0.60-0.75，PersonalPlan 目标 ≥ 0.97。
DC — Dependency Completeness rule-based · ∈ [0,1] · adapted from LRPLAN
是什么 / 评估目标
DC 测的是 plan 内 depends_on 字段里的引用是否真的指向 plan 内存在的 step。它不查环（那是 AR 的事），只查"悬空指针"——LLM 经常生成 depends_on: ["S99-9"] 但 plan 内根本没有 S99-9。DC 是连续指标（"多少比例的引用是合法的"），不像 SV/AR 是 0/1，所以能给出更细的渐进信号。
参考工作
等价于 LRPLAN \citep{karthikeyan2025lrplan} TravelPlanner / TimeArena 的 dependency-resolution rate（连续值版本）。Plan-quality 论文中的"reference integrity"系列指标都是同一族。
输入
plan 内所有 step 的 depends_on 列表 + 所有 step.id 集合。
计算公式
令 \(\mathcal{D}_{\mathrm{tot}}(\mathcal{P}) = \{(s, d) : s \in \mathcal{S}, d \in s.\mathrm{depends\_on}\}\) 是全部 plan 内 depends_on 引用集；\(\mathcal{D}_{\mathrm{sat}}(\mathcal{P}) = \{(s, d) \in \mathcal{D}_{\mathrm{tot}} : d \in \mathcal{S}\}\) 是指向 plan 内实际存在 step 的引用集。
\[ \mathrm{DC}(\mathcal{P}) \;=\; \begin{cases} \dfrac{|\mathcal{D}_{\mathrm{sat}}(\mathcal{P})|}{|\mathcal{D}_{\mathrm{tot}}(\mathcal{P})|}, & |\mathcal{D}_{\mathrm{tot}}(\mathcal{P})| > 0,\\[4pt] 1, & |\mathcal{D}_{\mathrm{tot}}(\mathcal{P})| = 0. \end{cases} \]
LaTeX 源 
\mathrm{DC}(\mathcal{P}) \;=\;
\begin{cases}
\dfrac{|\mathcal{D}_{\mathrm{sat}}(\mathcal{P})|}{|\mathcal{D}_{\mathrm{tot}}(\mathcal{P})|}, & |\mathcal{D}_{\mathrm{tot}}(\mathcal{P})| > 0,\\
1, & |\mathcal{D}_{\mathrm{tot}}(\mathcal{P})| = 0.
\end{cases}
参考代码
def dc(plan_json: dict) -> float:
    step_ids = {s["id"] for st in plan_json["output"]["subtasks"]
                for s in st["steps"]}
    total = satisfied = 0
    for st in plan_json["output"]["subtasks"]:
        for step in st["steps"]:
            for dep in step.get("depends_on", []):
                total += 1
                if dep in step_ids:
                    satisfied += 1
    return 1.0 if total == 0 else satisfied / total

方向 / 阈值

ATR — Agent-Tool Relevance rule-based · ∈ [0,1] · novel, extends iTool tool-binding check
是什么 / 评估目标
ATR 测的是 plan 内 agent / tool / subtask 三者的语义一致性——一个"SecurityAdvisor"agent 不应该被分配做"基础变量讲解"subtask；一个调用 ArxivPaperTool 的 step 不应该出现在"代码实现"型的 subtask 内。ATR 是三方匹配（agent↔subtask, tool↔agent, tool↔subtask），比单纯查"tool name 是否合法"（TBV）多了语义层面的判断。
参考工作
这是我们的扩展。Tool-binding 检查在 iTool \citep{zeng2025itool}（Parameter-Name/Value accuracy）和 BFCL \citep{patil2024bfcl}（Tool Call accuracy）里有现成版本，但都只查"tool 名是否在 agent.tools 里"——TBV 已经覆盖。ATR 额外加入 agent-role↔subtask-topic 的语义匹配，这层在 MAS 评估里尚无标准对应物，需要论文中明说："we introduce ATR as a novel metric for MAS plan evaluation, extending the tool-binding check of \citep{zeng2025itool} to the agent-role / subtask-topic dimension."
输入
plan 内所有 step（含 agent 和 tool 字段）+ agent list（含 tools）+ 一个 外部 mapping 表（role → recommended tools），约 30 行 YAML，覆盖典型角色（CodeDemonstrator / ConceptInstructor / SecurityAdvisor 等）。
计算公式
对每个 step \(s_j\)（\(j = 1, \dots, N\)），定义 3 个 0/1 indicator：
  • \(I^{(j)}_{A \to S}\): agent role 与 subtask 主题相关（rubric mapping，或 sentence embedding cosine \(> 0.3\)）
  • \(I^{(j)}_{T \to A}\): tool 在该 agent 的 tools 列表内（若 \(s_j.\mathrm{tool} = \varnothing\) 则置 0）
  • \(I^{(j)}_{T \to S}\): tool 适合该 subtask 类型（同上 mapping）
\[ \mathrm{ATR}(\mathcal{P}) \;=\; \frac{1}{N}\sum_{j=1}^{N} \frac{I^{(j)}_{A \to S} + I^{(j)}_{T \to A} + I^{(j)}_{T \to S}}{3} \]
LaTeX 源 
\mathrm{ATR}(\mathcal{P}) \;=\; \frac{1}{N}\sum_{j=1}^{N} \frac{I^{(j)}_{A \to S} + I^{(j)}_{T \to A} + I^{(j)}_{T \to S}}{3}
参考代码
ROLE_TOOLS = yaml.safe_load(open("role_tool_mapping.yaml"))

def atr(plan_json: dict) -> float:
    agents = {a["agent_role"]: a for a in plan_json["output"]["agents"]}
    total = 0.0; n = 0
    for st in plan_json["output"]["subtasks"]:
        for step in st["steps"]:
            ag = agents.get(step["agent"], {})
            i_as = int(any(kw in st["name"].lower()
                       for kw in ROLE_TOOLS.get(step["agent"], {}).get("subtask_kw", [])))
            if step.get("tool") is None:
                i_ta = i_ts = 0
            else:
                i_ta = int(step["tool"] in ag.get("tools", []))
                i_ts = int(step["tool"] in
                       ROLE_TOOLS.get(step["agent"], {}).get("recommended", []))
            total += (i_as + i_ta + i_ts) / 3; n += 1
    return 0.0 if n == 0 else total / n

TBV — Tool-Budget Validity (= 1 − Tool Hallucination Rate) rule-based · ∈ [0,1] · inverse of iTool tool-hallucination
是什么 / 评估目标
TBV 测的是"tool 是不是被合法 binding 的"——每个被 step 引用的 tool 必须：(1) 来自预定义 8-tool 池（防 hallucinate 工具名）；(2) 出现在该 step 所属 agent 的 tools 字段内（防 agent 越权调工具）。TBV 取反就是 Tool Hallucination Rate，是 tool-augmented agent 论文里的标准失败模式之一。
参考工作
等价于 iTool \citep{zeng2025itool}（Parameter-Name accuracy）和 BFCL v3 \citep{patil2024bfcl}（Tool Call accuracy）报告的 tool-binding pass rate 的取反——这两篇都把"tool 名错"作为 tool 错的主要失败模式。我们的 TBV 把"tool 在 pool 内"和"tool 在 agent.tools 内"做合取，相当于这两个条件的联合 hallucination rate。
输入
plan 全部 step.tool ≠ null 的 step + agent.tools 列表 + 预定义 8 工具池：{FirecrawlSearchTool, RagTool, CodeInterpreterTool, DirectoryReadTool, FileReadTool, FileWriterTool, CodeDocsSearchTool, ArxivPaperTool}。
计算公式
令 \(\mathcal{S}_{\mathrm{tool}} = \{s \in \mathcal{S} : s.\mathrm{tool} \ne \varnothing\}\)；\(\Pi\) 是预定义 8 工具池；\(\mathrm{ag}(s)\) 是 \(s\) 所属 agent。对每个 tool-bearing step \(s\)：
\[ v(s) \;=\; \mathbb{I}\!\left[\, s.\mathrm{tool} \in \Pi \,\right] \cdot \mathbb{I}\!\left[\, s.\mathrm{tool} \in \mathrm{ag}(s).\mathrm{tools} \,\right] \]
\[ \mathrm{TBV}(\mathcal{P}) \;=\; \begin{cases} \dfrac{1}{|\mathcal{S}_{\mathrm{tool}}|}\displaystyle\sum_{s \in \mathcal{S}_{\mathrm{tool}}} v(s), & |\mathcal{S}_{\mathrm{tool}}| > 0,\\[6pt] 1, & |\mathcal{S}_{\mathrm{tool}}| = 0. \end{cases} \]
LaTeX 源 
v(s) \;=\; \mathbb{I}\!\left[\, s.\mathrm{tool} \in \Pi \,\right] \cdot \mathbb{I}\!\left[\, s.\mathrm{tool} \in \mathrm{ag}(s).\mathrm{tools} \,\right]

\mathrm{TBV}(\mathcal{P}) \;=\;
\begin{cases}
\dfrac{1}{|\mathcal{S}_{\mathrm{tool}}|}\displaystyle\sum_{s \in \mathcal{S}_{\mathrm{tool}}} v(s), & |\mathcal{S}_{\mathrm{tool}}| > 0,\\
1, & |\mathcal{S}_{\mathrm{tool}}| = 0.
\end{cases}

TOOL_POOL = {"FirecrawlSearchTool", "RagTool", "CodeInterpreterTool",
             "DirectoryReadTool", "FileReadTool", "FileWriterTool",
             "CodeDocsSearchTool", "ArxivPaperTool"}

def tbv(plan_json: dict) -> float:
    agents = {a["agent_role"]: set(a.get("tools", []))
              for a in plan_json["output"]["agents"]}
    total = valid = 0
    for st in plan_json["output"]["subtasks"]:
        for step in st["steps"]:
            if step.get("tool") is None: continue
            total += 1
            if step["tool"] in TOOL_POOL and step["tool"] in agents.get(step["agent"], set()):
                valid += 1
    return 1.0 if total == 0 else valid / total

GED-sim — Structural Similarity to Gold rule-based · ∈ [0,1] · established (Sanfeliu & Fu)
是什么 / 评估目标
GED-sim 测的是 candidate plan 和 gold plan 在图结构层面的相似度——节点（step）集合和边（依赖）集合需要多少 insert/delete/substitute 操作能从一个变到另一个。这是论文里"plan 整体结构有多接近专家答案"的主结构指标，也是 GRPO 的 R_struct reward 里 StructSim 项的评估期对应物（训练用 multiset Jaccard 近似，评估用更严格的 GED）。
参考工作
Graph Edit Distance 是结构相似度的经典度量，最早由 Sanfeliu & Fu (1983) 提出 \citep{sanfeliu2012distance}；归一化 GED-sim 在 plan-quality 论文里被 PLAN-TUNING \citep{parmar2025plantuning} 的 \(R_{\text{plan}}\) reward 用作 candidate-vs-gold plan 相似度信号。NetworkX 提供标准实现 nx.graph_edit_distance()，可直接调用。
输入
candidate plan P 与 gold plan P★ 的依赖图 G' 和 G。节点 = step.id，边 = depends_on。
计算公式
令 \(G' = (\mathcal{S}', \mathcal{O}')\) 为 candidate 的依赖图，\(G^{\star} = (\mathcal{S}^{\star}, \mathcal{O}^{\star})\) 为 gold 的依赖图。\(\mathrm{GED}(\cdot, \cdot)\) 是节点 / 边 insert / delete / substitute 的最小总代价（substitute 用 label 0/1 cost）。
\[ \mathrm{GED\text{-}sim}(\mathcal{P}, \mathcal{P}^{\star}) \;=\; 1 - \frac{\mathrm{GED}(G', G^{\star})}{\max\!\bigl(|\mathcal{S}'| + |\mathcal{O}'|,\; |\mathcal{S}^{\star}| + |\mathcal{O}^{\star}|\bigr)} \]
LaTeX 源 
\mathrm{GED\text{-}sim}(\mathcal{P}, \mathcal{P}^{\star}) \;=\; 1 - \frac{\mathrm{GED}(G', G^{\star})}{\max\!\bigl(|\mathcal{S}'| + |\mathcal{O}'|,\; |\mathcal{S}^{\star}| + |\mathcal{O}^{\star}|\bigr)}
参考代码
import networkx as nx

def _build_graph(plan_json):
    G = nx.DiGraph()
    for st in plan_json["output"]["subtasks"]:
        for step in st["steps"]:
            G.add_node(step["id"], agent=step["agent"])
            for dep in step.get("depends_on", []):
                G.add_edge(dep, step["id"])
    return G

def ged_sim(plan, gold) -> float:
    G1, G2 = _build_graph(plan), _build_graph(gold)
    # timeout 2s 防止指数爆炸
    ged = nx.graph_edit_distance(G1, G2, timeout=2.0) or 0
    denom = max(G1.number_of_nodes() + G1.number_of_edges(),
                G2.number_of_nodes() + G2.number_of_edges())
    return 1.0 if denom == 0 else max(0.0, 1.0 - ged / denom)
实测注意：当 plan ≥ 15 step 时 GED 可能超 2 s timeout，此时改用近似算法 nx.optimize_graph_edit_distance(G1, G2) 取首个 yield 值。
✅ 通过示例（GED-sim 高）
gold plan 有 8 个 step、9 条依赖；candidate 也有 8 个 step、相同的依赖图骨架，只是 2 个 step 的 ID 命名不同（S2-1 vs S2-A），label substitution cost 2，无 insert / delete。
\(\mathrm{GED} = 2\)，\(\max(|\mathcal{S}|+|\mathcal{O}|) = 17\)，GED-sim = \(1 - 2/17 ≈ 0.88\)——结构骨架几乎一致。
❌ 不通过示例（GED-sim 低）
gold plan 有 8 个 step、9 条依赖；candidate 直接一刀切成 3 个并列 step、无依赖（典型 "prompt-and-solve" baseline 输出）。
\(\mathrm{GED} = 14\)（删 5 个 node + 删 9 条 edge）；max-denominator = 17，GED-sim = \(1 - 14/17 ≈ 0.18\)——候选 plan 跟 gold 几乎没有共同结构。这种情况说明 candidate 模型放弃了 multi-agent decomposition。
方向 / 阈值
↑ higher better。SOTA ~0.35-0.55（即便 frontier LLM 也只能"大概像"gold），弱 baseline ~0.15-0.30，PersonalPlan 目标 ≥ 0.55-0.65（GRPO 的 StructSim reward 直接优化它）。

11.2 Tier 1 LLM-judge plan-quality 指标（4 项）

Pers. — Personalization LLM judge · 3 sub-dim · ∈ [0,1] · KELE-style anchored rubric
是什么 / 评估目标
Pers. 测的是 plan 在"个性化"维度上的语义质量——profile 是真的影响了 plan 的起点（skill 匹配）、方向（goal 对齐）、例子（背景适配），还是只是把 profile 当装饰？这是 RQ1（"PersonalPlan 是否真的 personalized"）的主判别指标。三个子维度独立 LLM call，防止 halo effect（一个维度高把别的都拉高）。
参考工作
5/3/1 anchored Likert 子维度独立打分的 rubric 设计来自 KELE \citep{peng2025kele}（PRR/NDAR/SPR/IAR 用同样架构）和 Personality-aware Student Simulation \citep{liu2024personalitysim}（cognitive + noncognitive 多维度独立 categorisation）。"为个性化教学评分"的具体子维度（Skill / Goal / Background）是我们针对编程教育场景设计的，但 rubric 形式照搬 KELE。
输入
(query q, profile I_p, plan P) 三元组。每个子维度独立调用 1 次 LLM。
计算公式
令 \(r_{\mathrm{sm}}, r_{\mathrm{go}}, r_{\mathrm{ba}} \in \{1, 3, 5\}\) 为 3 个子维度（SkillMatch / GoalOrientation / BackgroundAdaptation）的 anchored Likert 评分；\(J\) 为 LLM judge。
\[ \bar{r}_{\mathrm{Pers}}(\mathcal{P}, I_p) \;=\; \frac{1}{3}\!\left( r_{\mathrm{sm}} + r_{\mathrm{go}} + r_{\mathrm{ba}} \right), \qquad r_{(\cdot)} \sim J(\mathcal{P}, I_p) \]
\[ \mathrm{Pers.}(\mathcal{P}, I_p) \;=\; \frac{\bar{r}_{\mathrm{Pers}}(\mathcal{P}, I_p) - 1}{4} \;\in\; [0, 1] \]
LaTeX 源 
\bar{r}_{\mathrm{Pers}}(\mathcal{P}, I_p) \;=\; \frac{1}{3}\!\left( r_{\mathrm{sm}} + r_{\mathrm{go}} + r_{\mathrm{ba}} \right)

def pers(plan, query, profile, judge="gpt-5") -> float:
    sm = llm_judge_score(SKILL_MATCH_RUBRIC, plan, query, profile, judge)

Feas. — Feasibility LLM judge · 3 sub-dim · ∈ [0,1] · KELE-style + reference-grounded (Aligning-Pedagogy)
是什么 / 评估目标
Feas. 测的是 plan 作为"可执行教学指南"在逻辑层是否合理——依赖顺序对不对、任务分解粒度合不合适、agent / tool 配对是否符合常识。这不是单查 schema（SV）或图结构（AR/DC），而是语义层面的"reviewer 看了点不点头"。Feas. 与 Tier 2 的 EVR（真跑一遍）互补——Feas. 廉价但 LLM-judged，EVR 贵但客观。
参考工作
Rubric 形式同 Pers.，来自 KELE \citep{peng2025kele}。关键技巧："reference-grounded judging"——把 gold plan 的 execution_order 作为 reference 给 judge 看，让它判 candidate "是否同样合理"，比 zero-shot "判这个 plan 好不好"准很多。这个技巧来自 Aligning-Pedagogy \citep{dinucu2025aligningpedagogy} 的 reference-aware judge prompt 设计。
输入
(query, plan P, gold plan P★ 的 execution_order 作 reference) 三元组。reference-grounded judging。
计算公式
令 \(r_{\mathrm{do}}, r_{\mathrm{td}}, r_{\mathrm{at}} \in \{1, 3, 5\}\) 为 3 个子维度（DepOrdering / TaskDecomp / AgentToolApprop）的 anchored Likert 评分，judge 在 prompt 中可获得 gold plan 的 execution_order 作 reference。
\[ \mathrm{Feas.}(\mathcal{P} \mid \mathcal{P}^{\star}) \;=\; \frac{1}{4}\!\left( \frac{r_{\mathrm{do}} + r_{\mathrm{td}} + r_{\mathrm{at}}}{3} - 1 \right) \]
LaTeX 源 
\mathrm{Feas.}(\mathcal{P} \mid \mathcal{P}^{\star}) \;=\; \frac{1}{4}\!\left( \frac{r_{\mathrm{do}} + r_{\mathrm{td}} + r_{\mathrm{at}}}{3} - 1 \right)
参考代码
def feas(plan, query, gold_plan, judge="gpt-5") -> float:

Ped. — Pedagogy = mean(PRR, NDAR, SPR, IAR) judge × 3 + rule × 1 · ∈ [0,1] · 4 sub-metrics ported from KELE
是什么 / 评估目标
Ped. 测的是 plan 在"教学法"维度上的整体质量——是否符合学习规律（PRR）、是否不直接给答案（NDAR）、是否覆盖完整的教学阶段（SPR）、风格是否适配 profile（IAR）。这是论文里 R_ped reward 在评估期的对应指标，没有 Ped. 这一项的话 RQ3（"R_ped 的 ablation 贡献"）就没法报。
参考工作
这 4 个子指标 PRR / NDAR / SPR / IAR 直接 port 自 KELE \citep{peng2025kele}——KELE 用同样 4 项评估 Socratic 教学 plan，文中报 100 条人工 \(\kappa\) 0.65-0.75。我们的扩展仅在于把 KELE 的 5 个 Socratic phase 换成 6 个编程教学 phase（probe / retrieve-demonstrate / apply / validate / feedback / consolidate），其余 rubric 完全照搬。这意味着 Ped. 不需要论文里花篇幅辩护其设计——直接 cite KELE。
输入
(query, profile, plan, accepted_answer) 四元组。3 个 LLM call + 1 个规则式 phase 检测。
计算公式
4 个子项：
  • \(r_{\mathrm{PRR}} \in \{1,3,5\}\)：subtask 顺序是否符合学习规律（LLM judge）
  • \(\mathrm{NDAR}(\mathcal{P}) \in [0,1]\)：plan 不在前期直接给 accepted answer 的"程度"（LLM judge，反向归一化）
  • \(\mathrm{SPR}(\mathcal{P}) \in [0,1]\)：6 个 instructional phase \(\Phi = \{\text{probe, retr-demo, apply, validate, feedback, consolidate}\}\) 的覆盖率（规则式）
  • \(r_{\mathrm{IAR}} \in \{1,3,5\}\)：inductive / deductive 风格是否匹配 profile（LLM judge）
\[ \mathrm{SPR}(\mathcal{P}) \;=\; \frac{1}{|\Phi|} \sum_{\phi \in \Phi} \mathbb{I}\!\left[\, \exists\, s \in \mathcal{S}:\; \mathrm{phase}(s) = \phi \,\right] \]
\[ \mathrm{Ped.}(\mathcal{P}, I_p) \;=\; \frac{1}{4}\!\left( \frac{r_{\mathrm{PRR}} - 1}{4} + \mathrm{NDAR}(\mathcal{P}) + \mathrm{SPR}(\mathcal{P}) + \frac{r_{\mathrm{IAR}} - 1}{4} \right) \]

def ped(plan, query, profile, accepted_answer, judge="gpt-5") -> float:
    prr = llm_judge_score(PRR_RUBRIC, plan, query, None, judge)
    ndar = llm_judge_ndar(plan, accepted_answer, judge)  # 返回 [0,1]
    spr = sum(any(kw in step["instruction"].lower()
                  for st in plan["output"]["subtasks"] for step in st["steps"]
                  for kw in kws)
              for kws in PHASE_KEYWORDS.values()) / 6
    iar = llm_judge_score(IAR_RUBRIC, plan, query, profile, judge)

Sati. — Profile-Conditioned Win Rate vs Gold (Tier 3) LLM pairwise · M-judge product · ∈ [0,1] · RLHF win rate × Aligning-Pedagogy M-judge
是什么 / 评估目标
Sati. 测的是"从 learner 视角看，candidate plan 比 gold plan 更好的程度"。这是 Tier 3 的唯一主指标，也是 RQ1 最直接的人本判别——它把"plan 能不能让 learner 满意"压缩成一个数。Sati. 的两个独特设计：(1) 不告 judge 哪个是 gold，避免 anchoring bias；(2) AB / BA 两次都必须一致才算稳定，否则算 Tie，防止 position bias。
参考工作
Pairwise win-rate 是 RLHF 的标准 evaluation paradigm（\citep{ouyang2022instructgpt} 等），M-judge 乘积聚合则来自 Aligning-Pedagogy \citep{dinucu2025aligningpedagogy}——他们在 pedagogy RM 上用 M 个独立 judge 的乘积而非平均，因为乘积对单 judge 的 reward hacking 更敏感。我们这里把两者结合：每个 judge 算 win rate（RLHF 标准），3 个 judge 取乘积（Aligning-Pedagogy 标准）。
输入
每个 test query 一对 plan：(candidate P', gold P★) + learner profile。不告知 judge 哪个是 gold。每对要跑 AB 和 BA 两次。
计算公式
对 test 集第 \(i\) 个 query，candidate plan \(\mathcal{P}'_i\) 与 gold plan \(\mathcal{P}^{\star}_i\) 在 judge \(m\) 下、顺序 \(o \in \{\mathrm{AB}, \mathrm{BA}\}\) 的偏好结果：
\[ y^{(m, o)}_i \;\in\; \{\mathrm{cand}, \mathrm{gold}, \mathrm{tie}\}, \qquad i = 1, \dots, N \]
单对得分（取 AB 和 BA 一致为准，矛盾计为 tie）：
\[ \pi^{(m)}_i \;=\; \frac{1}{2}\!\left( \mathbb{I}[y^{(m, \mathrm{AB})}_i = \mathrm{cand}] + \mathbb{I}[y^{(m, \mathrm{BA})}_i = \mathrm{cand}] \right) + \frac{1}{2}\,\mathbb{I}[y^{(m, \mathrm{AB})}_i \ne y^{(m, \mathrm{BA})}_i] \]
单 judge 偏好率与 M-judge 乘积聚合（仿 Aligning-Pedagogy）：
\[ w_m \;=\; \frac{1}{N}\sum_{i=1}^{N} \pi^{(m)}_i, \qquad \mathrm{Sati.} \;=\; \prod_{m=1}^{M} w_m \]
LaTeX 源 
\pi^{(m)}_i \;=\; \frac{1}{2}\!\left( \mathbb{I}[y^{(m, \mathrm{AB})}_i = \mathrm{cand}] + \mathbb{I}[y^{(m, \mathrm{BA})}_i = \mathrm{cand}] \right) + \frac{1}{2}\,\mathbb{I}[y^{(m, \mathrm{AB})}_i \ne y^{(m, \mathrm{BA})}_i]

w_m \;=\; \frac{1}{N}\sum_{i=1}^{N} \pi^{(m)}_i, \qquad
\mathrm{Sati.} \;=\; \prod_{m=1}^{M} w_m
为什么用乘积而非平均：一个 judge 不喜欢 → 整体掉很多（仿 Aligning-Pedagogy）；平均会被高 judge 拉起来导致 reward hacking。
参考代码
JUDGES = ["openai/gpt-5", "anthropic/claude-opus-4.6", "google/gemini-3-pro"]

def sati_pair(cand, gold, profile, judge) -> float:
    ab = call_pairwise_judge(judge, cand, gold, profile, order="AB")  # 'A'|'B'|'Tie'
    ba = call_pairwise_judge(judge, gold, cand, profile, order="BA")
    # AB 选 A = 选 cand；BA 选 B = 选 cand
    cand_picks = (ab == "A") + (ba == "B")
    ties = (ab == "Tie") + (ba == "Tie")
    if ab != ba and "Tie" not in (ab, ba):  # 矛盾 → 计 Tie
        return 0.5
    return (cand_picks + 0.5 * ties) / 2

def sati(test_pairs, profiles, judges=JUDGES) -> float:
    per_judge = []
    for j in judges:
        wins = [sati_pair(c, g, p, j) for (c, g), p in zip(test_pairs, profiles)]
        per_judge.append(sum(wins) / len(wins))
    return math.prod(per_judge)
✅ 通过示例（Sati. 高）
candidate plan 对一个 Python beginner 的 learner，从"先问你知道 list 是什么吗"开始，每步用 learner 在 about_me 里提到的"flask web 后端"作类比；gold plan 是 generic 的"从 list comprehension 讲起"。3 个 judge 都偏好 candidate：\(w_{\mathrm{GPT5}} = 0.78\), \(w_{\mathrm{Claude}} = 0.82\), \(w_{\mathrm{Gemini}} = 0.74\) → Sati. = 0.78 × 0.82 × 0.74 ≈ 0.47。
❌ 不通过示例（Sati. 低）
candidate plan 跟 gold 看起来差不多但 step 描述简略、用了不相关的例子。3 个 judge 多数偏好 gold：\(w = 0.30, 0.35, 0.28\) → Sati. = 0.30 × 0.35 × 0.28 ≈ 0.029。
极端反面：candidate 在 judge GPT-5 看着不错（\(w=0.65\)）但 Claude 觉得很差（\(w=0.20\)）——乘积 0.65 × 0.20 × ... 会被 Claude 这一项整体压低，这就是 M-judge product 的设计意图：防止"在一个 judge 上 hack 高分"。
方向 / 阈值
↑ higher better。完全打平 GT → 单 judge 0.5，3-judge 乘积 0.125。SOTA ~0.20-0.30，PersonalPlan 目标 ≥ 0.45（即 candidate 在 3 个 judge 上各自 ≥0.77 偏好率才能达到）。
评分标准（pairwise judge prompt）

Output exactly one of: "A", "B", or "Tie". Tie ONLY if both are clearly equivalent.

User:
Query: {query}
Learner profile: {profile}

Plan A: {plan_A_json}
Plan B: {plan_B_json}

  1. Skill Match: which plan starts at a level appropriate to my declared skills?
  2. Engagement & Learnability: which is easier to follow with clearer instructions?
  3. Structural Appropriateness: which has a more logical structure I can actually execute?

判定细则：(a) AB / BA 必须分别跑；(b) AB 和 BA 结果必须一致才算稳定，矛盾计为 Tie；(c) 跨 judge 乘积；(d) 不告 judge 哪个是 gold；(e) plan JSON 进 prompt 前 sanitize 删除可能的 prompt injection（如出现"please pick this plan"等字串就剔除）。

11.3 Tier 1 反事实探针（1 项 + 配套的 PNG）

PVS — Profile Sensitivity (Profile-Variance Score) rule-based aggregate · ∈ [0,1] · novel; adapted from MSI-Agent OOD probe + Personality-Sim with/without comparison
是什么 / 评估目标
PVS 测的是"profile 信号是否真的影响了 plan generation"。对同一个 query，给不同 profile 应该产生不同的 plan；如果所有 profile 下生成的 plan 几乎一样（PVS ≈ 0），说明 PAD 没起作用，profile 只是装饰。PVS 是 RQ2（"PersonalPlan 是否随 profile 适配"）的必备 sanity check——没有它就无法反驳"reviewer：你们的 plan 真的随 profile 变了吗？"。
参考工作
这是我们引入的新指标，但思想来自两个 prior art：(1) MSI-Agent \citep{fu2024msiagent} kitchen→livingroom→bedroom 的 OOD shift probe——同一 method 在不同环境下生成行为差异度；(2) Personality-aware Student Sim \citep{liu2024personalitysim} with-vs-without BF-TC 对比，分析 personality 信号是否真的影响生成的对话。论文里需要明说："we adapt the OOD-distribution-shift diagnostic of \citet{fu2024msiagent} to profile-conditioned plan generation."
输入
对 test 中每个 ≥2 profile variants 的 query q（共 97 个），用同一 model 生成 k 个 plan（一个 profile 一个）。复用 GED-sim 的图构造。
计算公式
令 \(\mathcal{Q}_{\ge 2}\) 是 test 中 profile 数 \(k_q \ge 2\) 的 query 集合（实测 \(|\mathcal{Q}_{\ge 2}| = 97\)）；对每个 \(q \in \mathcal{Q}_{\ge 2}\)，model 在 \(k_q\) 个 profile 下分别生成 plan \(\{\mathcal{P}_q^{(1)}, \dots, \mathcal{P}_q^{(k_q)}\}\)。
\[ \mathrm{PVS}(q) \;=\; 1 - \binom{k_q}{2}^{\!-1} \sum_{1 \le i < j \le k_q} \mathrm{GED\text{-}sim}\!\left(\mathcal{P}_q^{(i)}, \mathcal{P}_q^{(j)}\right) \]
\[ \mathrm{PVS} \;=\; \frac{1}{|\mathcal{Q}_{\ge 2}|} \sum_{q \in \mathcal{Q}_{\ge 2}} \mathrm{PVS}(q) \;\in\; [0, 1] \]
LaTeX 源 
\mathrm{PVS}(q) \;=\; 1 - \binom{k_q}{2}^{\!-1} \sum_{1 \le i < j \le k_q} \mathrm{GED\text{-}sim}\!\left(\mathcal{P}_q^{(i)}, \mathcal{P}_q^{(j)}\right)

\mathrm{PVS} \;=\; \frac{1}{|\mathcal{Q}_{\ge 2}|} \sum_{q \in \mathcal{Q}_{\ge 2}} \mathrm{PVS}(q)
高 PVS = plan 真的随 profile 改变；低 PVS = plan 是同一模板。
参考代码
from itertools import combinations

def pvs(model_gen_fn, test_qids_with_profiles) -> float:
    # test_qids_with_profiles: [(qid, [profile1, profile2, ...]), ...]
    pvs_per_q = []
    for qid, profiles in test_qids_with_profiles:
        if len(profiles) < 2: continue
        plans = [model_gen_fn(qid, p) for p in profiles]
        sims = [ged_sim(plans[i], plans[j])
                for i, j in combinations(range(len(plans)), 2)]
        pvs_per_q.append(1.0 - sum(sims) / len(sims))
    return sum(pvs_per_q) / len(pvs_per_q)
方向
↑ higher better（指明 plan 在变化）。但太高也不对——如果 PVS > 0.7 说明 model 对同一 query 出完全不同的 plan（结构不稳定）。
预期：
  • PersonalPlan: 0.35–0.50（结构骨架共享，profile-specific 部分变化）
  • w/o PAD baseline: 0.05–0.10（plan 几乎不变 → PAD 没用）
  • Random baseline: ~0.70+（plan 完全不稳定）
论文里报这个值同时配合 Pers. 看：高 PVS + 高 Pers. = 真的个性化；高 PVS + 低 Pers. = 乱变。
✅ 通过示例（PVS 高）

PersonalPlan 生成 3 个 plan，骨架类似（5-6 subtask），但 step 内容/agent role/示例代码全部不同：plan (a) 用 @Cacheable 注解，plan (b) 用 flask-caching，plan (c) 用 node-cache。两两 GED-sim ≈ 0.55-0.65 → PVS(q) = 1 − 0.6 ≈ 0.40。

同 query，但 baseline 模型对 3 个 profile 生成几乎一样的 plan——只是 step 末尾加了一句"considering you use Java/Python/Node"。两两 GED-sim ≈ 0.94 → PVS(q) = 1 − 0.94 ≈ 0.06。说明 baseline 没真正用 profile 去 condition decomposition，只是 superficial mention。
PNG — Counterfactual Personalization Gap LLM judge aggregate · ∈ [-1,1] · novel; adapted from Personality-Sim with/without comparison
是什么 / 评估目标
PNG 测的是"用真实 profile 生成的 plan 是不是真比用随机 profile 生成的 plan 更适配该 learner"。这是 PVS 的第二支柱：PVS 测"plan 变没变"，PNG 测"变的方向对不对"。两者必须同时高才能 confirm personalization is real。
参考工作
"换 profile 看效果"的对照实验法源自 Student-Data Paradox \citep{sonkar2024studentparadox} 的 student-shuffled control 与 Personality-Sim \citep{liu2024personalitysim} 的 with/without BF-TC 对比。PNG 把这种 with-vs-shuffled 的 gap 量化为单个数。论文里说："we report PNG as a counterfactual probe analogous to the shuffled-profile control of \citet{liu2024personalitysim}."
输入
同一 query 跑两次：(a) with real profile；(b) with profile-shuffled（随机抽 test 集另一 profile 顶上）。
计算公式
对每个 (query, profile) 对 \((q_i, I_p^{(i)})\)，令 \(\tilde{I}_p^{(i)}\) 是从 test 集其他 profile 中均匀抽样得到的 shuffled profile（\(\tilde{I}_p^{(i)} \ne I_p^{(i)}\)）；\(\mathrm{gen}(q, I_p)\) 是 model 生成函数。
\[ \mathrm{PNG} \;=\; \frac{1}{N}\sum_{i=1}^{N} \Bigl[\mathrm{Pers.}\!\bigl(\mathrm{gen}(q_i, I_p^{(i)}),\, I_p^{(i)}\bigr) - \mathrm{Pers.}\!\bigl(\mathrm{gen}(q_i, \tilde{I}_p^{(i)}),\, I_p^{(i)}\bigr) \Bigr] \;\in\; [-1, 1] \]
LaTeX 源 
\mathrm{PNG} \;=\; \frac{1}{N}\sum_{i=1}^{N} \Bigl[\mathrm{Pers.}\!\bigl(\mathrm{gen}(q_i, I_p^{(i)}),\, I_p^{(i)}\bigr) - \mathrm{Pers.}\!\bigl(\mathrm{gen}(q_i, \tilde{I}_p^{(i)}),\, I_p^{(i)}\bigr) \Bigr]
注意：判 Pers. 时仍然用真实 profile \(I_p^{(i)}\) 作 ground truth，只是生成时用 shuffled。PNG 显著正 → 个性化真的依赖 profile；PNG ≈ 0 → profile 只是装饰。
参考代码
def png(model_gen_fn, test_qids_with_profiles, shuffle_rng=42) -> float:
    rng = random.Random(shuffle_rng)
    all_profiles = [p for _, ps in test_qids_with_profiles for p in ps]
    real_pers, shuffled_pers = [], []
    for qid, profiles in test_qids_with_profiles:
        for p in profiles:
            shuf = rng.choice([x for x in all_profiles if x != p])
            real_pers.append(pers(model_gen_fn(qid, p), qid_query(qid), p))
            shuffled_pers.append(pers(model_gen_fn(qid, shuf), qid_query(qid), p))
    return mean(real_pers) - mean(shuffled_pers)

profile="senior Java dev"。PersonalPlan 用真实 profile 生成的 plan 评 Pers.=0.78；用 shuffled profile（替换成"complete beginner"）生成的 plan 评 Pers.=0.32（因为"complete beginner" plan 用在 senior dev 身上完全不合适）。
PNG = 0.78 − 0.32 = +0.46，paired t-test p < 0.001 → profile 信号真的驱动了 plan 的 personalization 方向。
❌ 不通过示例（PNG ≈ 0）
baseline 模型对真实 profile 和 shuffled profile 都生成差不多 generic的 plan：Pers.(real) = 0.42, Pers.(shuffled) = 0.40。
PNG = 0.42 − 0.40 = +0.02，p = 0.45 → profile 对生成几乎无影响，"个性化"只是表面装饰，model 实际无视 profile。
方向 / 阈值
↑ higher better。PersonalPlan 目标 ≥ +0.15（绝对差距），baseline 多数 ≈ 0。
显著性检验：用 paired t-test 在 N=305 上检验 PNG > 0（p < 0.01）；论文里报 p-value。

11.4 Tier 2 — Axis A: Plan 执行可行性（1 项 headline）

v1.1 修订说明（2026-05-19，dry-run 后的设计修复）
v1.0 设计在 MAP-PPL-Test dry-run 中跑出 EVR ≡ 0，根因是两个操作化假设在 MAP-PPL 数据上不成立：
  1. 原 (e) out / CCR 让 LLM judge 比对真实 stdout 与 plan 字段 expected_output 是否功能等价。但 MAP-PPL 的 expected_output 在数据集构造时就是描述性散文（如"Documented exception classes: FileNotFoundError / OSError from open(), json.JSONDecodeError from json.load"），不是 literal stdout，judge 必然判 ≠ → CCR=0 几乎必然。
  2. 原 (d) exec / CER 用排除式关键词扫描（"Expected output:" / "should see:" 等 6 种臆测 pattern）来判定"代码真跑 vs 假跑"。但 MAP-PPL 上 agent 真实失败模式是干脆不调 tool、直接给一段干净教学文字——这段文字不含任何臆测关键词，反而被误判为"真执行"，把不可信内容传给下游 CCR 判定。
v1.1 修复：把 exec 和 out 都改为对客观 execution trace 的纳入式检测：
  • exec（CER）：检查 execution_log[step_id].tool_calls 中是否存在 tool == "CodeInterpreterTool" 的 invocation event。没调 tool 就是 0，跟措辞无关。
  • out（CCR）：检查 sandbox 的 exit_code == 0 且无 unhandled exception。这正是 v1.0 已经引的 WebAgent-R1 program-exec binary reward 的原始定义，反而比"功能等价 judge"更贴 cite。
  • 原 LLM 功能等价 judge 不丢弃，降级到 appendix B 作 fuzzy 互补信号——只在 plan 给出了结构化 output_assertion 字段时启用，对纯散文 expected_output 不报。
EVR 5-项 AND 的 headline 结构、prior-art 引用、headline 地位均不变；变的只是 (d)(e) 两个子检查的可操作定义。
EVR — Execution Validity Rate aggregate · headline · Q1 · ∈ [0,1] · ≈ AgentGym task success + KELE WC + WebAgent-R1 program-exec 联合 · v1.1
是什么 / 评估目标
EVR 直接回答"plan 真的能被 CrewAI 正确执行吗？"——5 个 binary 条件 AND 联合 PASS 的 run 比例：(a) 所有 step 跑完非空；(b) 所有 loop 都有 exit_reason 且与最后一轮内容自洽；(c) 所有 depends_on 上游 step 都有非空有用输出；(d) 所有声明 CodeInterpreterTool 的 step 在 CrewAI execution log 里有真实 tool 调用事件（纳入式：读 tool_calls 字段，不靠关键词排除）；(e) 所有真调用的 code step 在 sandbox 中执行成功（exit_code == 0 且无 unhandled exception）。EVR 是 Tier 2 唯一与论文 abstract "在 X% test query 上跑通了端到端教学"对应的那个 X。
参考工作
EVR 直接套用 3 个 prior-art 范式：(i) AgentGym \citep{xi2025agentgym} 的 task success rate（多 env 联合 PASS 聚合）；(ii) WebAgent-R1 \citep{wei2025webagentr1} 的 program-exec binary reward（"代码真跑了 vs 假跑了"，v1.1 后 CCR 直接等价于这个定义）；(iii) 对 tool-call 是否真实发生的"纳入式"检测对标 ToolBench \citep{qin2023toolbench} 的 tool invocation 审计。WC 3 子检查（cov / loop / flow）直接来自 KELE \citep{peng2025kele} workflow check。这种"工作流 3 子检查 ∧ tool-call ∧ exec-success"的多条件 AND 聚合在 LRPLAN \citep{karthikeyan2025lrplan} TravelPlanner final pass-rate (commonsense ∧ hard) 中已经被使用。
输入
(plan JSON, execution_log.json, step_outputs.json) 三元组。execution_log 由 PlanRuntime + CrewAI live mode 产出，含每个 step 的 actual_interaction、teacher_output、student_response、tool_call_log；step_outputs 含 code step 的 sandbox stdout/stderr + 检测到的 language。
计算公式 v1.1
对 run \(r\)，令 \(\mathcal{S}_r\) 为该 plan 的全部 step，\(\mathcal{S}^{\mathrm{code}}_r \subseteq \mathcal{S}_r\) 为声明 tool=CodeInterpreterTool 的 step（来自 plan JSON），\(\mathcal{S}^{\mathrm{real}}_r \subseteq \mathcal{S}^{\mathrm{code}}_r\) 为在 CrewAI execution log 中实际发生过 CodeInterpreterTool 调用事件的 step（即 exec 检测通过的子集）。定义 5 个 binary sub-check（\(\mathrm{check}_c(r) \in \{0, 1\}\)，全部由现有 evaluator.py 或 ~50 行 CCR/CER 脚本得到）： \[ \begin{aligned} \mathrm{cov}_r &\;=\; \mathbb{I}[\,\forall s \in \mathcal{S}_r:\; \mathrm{actual\_interaction}(s) \ne \emptyset \,]\\ \mathrm{loop}_r &\;=\; \mathbb{I}[\,\forall \mathrm{loop} \in \mathcal{P}_r:\; \mathrm{exit\_reason} \ne \emptyset \,\wedge\, \mathrm{reason\_consistent}\,]\\ \mathrm{flow}_r &\;=\; \mathbb{I}[\,\forall s' \in \mathrm{depends\_on}(s):\; \mathrm{output}(s') \ne \emptyset \,\wedge\, \mathrm{useful} \,]\\ \mathrm{exec}_r &\;=\; \mathbb{I}\!\left[\forall s \in \mathcal{S}^{\mathrm{code}}_r:\; \exists\, tc \in \mathrm{tool\_calls}(s),\ tc.\mathrm{name} = \text{CodeInterpreterTool} \right] \quad \text{(即 CER}_r = 1\text{)}\\ \mathrm{out}_r &\;=\; \mathbb{I}\!\left[\forall s \in \mathcal{S}^{\mathrm{real}}_r:\; \mathrm{exit\_code}(s) = 0 \,\wedge\, \neg\,\mathrm{unhandled\_exception}(s)\right] \quad \text{(即 CCR}_r = 1\text{)} \end{aligned} \] 约定：若 \(\mathcal{S}^{\mathrm{code}}_r = \emptyset\)（plan 完全没有 code step，例如纯 Socratic 对话型 plan），\(\mathrm{exec}_r\) 与 \(\mathrm{out}_r\) 都按 vacuously true 取 1，EVR 自然退化为 \(\mathrm{cov}_r \cdot \mathrm{loop}_r \cdot \mathrm{flow}_r\)，避免空分母伪 0 / 伪 1。 Run-level verdict（5 个全 PASS）：
\[ \mathrm{EVR\text{-}PASS}_r \;=\; \mathrm{cov}_r \cdot \mathrm{loop}_r \cdot \mathrm{flow}_r \cdot \mathrm{exec}_r \cdot \mathrm{out}_r \]
聚合：
\[ \mathrm{EVR} \;=\; \frac{1}{|\mathcal{R}|}\sum_{r \in \mathcal{R}} \mathrm{EVR\text{-}PASS}_r \;\in\; [0, 1] \]
LaTeX 源 
\mathrm{EVR\text{-}PASS}_r \;=\; \mathrm{cov}_r \cdot \mathrm{loop}_r \cdot \mathrm{flow}_r \cdot \mathrm{exec}_r \cdot \mathrm{out}_r

\mathrm{EVR} \;=\; \frac{1}{|\mathcal{R}|}\sum_{r \in \mathcal{R}} \mathrm{EVR\text{-}PASS}_r
注意 \(\mathrm{EVR} \le \min(\mathrm{cov}, \mathrm{loop}, \mathrm{flow}, \mathrm{CER}, \mathrm{CCR})\)，是 Tier 2 最严的指标。

TOOL_NAME = "CodeInterpreterTool"

def _declared_code_steps(plan):
    """从 plan JSON 抽出声明 tool=CodeInterpreterTool 的 step。"""
    out = []
    for sub in plan["output"]["subtasks"]:
        for step in sub["steps"]:
            if step.get("tool") == TOOL_NAME:
                out.append(step["id"])
    return out

def _tool_invoked(exec_log, step_id):
    """v1.1 纳入式检测：execution_log 中该 step 是否真的调用了 CodeInterpreterTool。"""
    tcs = exec_log.get(step_id, {}).get("tool_calls", [])
    return any(tc.get("name") == TOOL_NAME for tc in tcs)

def evr_run(plan, exec_log, step_outputs) -> int:
    # (a)(b)(c) 复用 KELE WC 三个子检查
    report = evaluate_execution(plan, exec_log)
    cov   = "Check 2a: Step Coverage" in report and "PASS" in _slice(report, "2a", "2b")
    loop  = "Check 2b: Loop Behavior" in report and "PASS" in _slice(report, "2b", "2c")
    flow  = "Check 2c: Information Flow" in report and "PASS" in _slice(report, "2c", "2d")

    code_ids = _declared_code_steps(plan)
    # (d) CER：所有声明 CodeInterpreterTool 的 step 都必须有 tool_call event
    real_ids = [sid for sid in code_ids if _tool_invoked(exec_log, sid)]
    exec_ok  = (len(real_ids) == len(code_ids))    # 没声明 code step 时 vacuously True

    # (e) CCR：所有真调用的 code step sandbox 执行成功
    out_ok   = all(_sandbox_ok(step_outputs, sid) for sid in real_ids)

    return int(cov and loop and flow and exec_ok and out_ok)

def evr(runs) -> float:
    return sum(evr_run(*r) for r in runs) / len(runs)

PAS — Personalization Alignment Score LLM judge per utterance · headline · Q2.a · ∈ [0,1] · = KELE PRR (Personalization Reflection Rate)
是什么 / 评估目标
PAS 测"MAS 教学过程中，教师 agent 实际说的话是否反映 learner_profile"——profile 字段包括 age、knowledge_level、learning_style（visual / kinesthetic / etc.）、native_language、interests。PAS 的判定对象是每条 teacher utterance（不是整个 plan），因此能捕捉到"plan 写了适配 K12 学生，但 runtime 教师 agent 还是用 PhD-level 术语"这种 plan-execution 脱节。
参考工作
PAS 直接采用 KELE \citep{peng2025kele} 的 PRR (Personalization Reflection Rate)，论文里建议直接命名为 "Personalization Reflection Rate (PRR)" + cite KELE，不用我们的 PAS 缩写。Profile-conditioned generation evaluation 的另一个 anchor 是 Personality-Sim \citep{liu2024personalitysim} 的 BF-TC 5/3/1 anchored Likert——我们的 PAS 用同样 3-档 score。与论文 §3.5 MAP-PPL 数据 profile 7 字段一致。
输入
(learner_profile JSON, execution_log) — profile 来自 plan.input.learner；execution_log 提供每个 step 的 teacher_output（含 loop iterations）。
计算公式
令 \(\mathcal{U}_r^{\mathrm{tch}}\) 为 run \(r\) 的全部 teacher utterance；对每个 \(u \in \mathcal{U}_r^{\mathrm{tch}}\)，judge \(J\) 输出 alignment score \(p_u \in \{2, 1, 0\}\)（well-aligned / partially aligned / not aligned）。归一到 [0,1]：
\[ \mathrm{PAS}_r \;=\; \frac{1}{2\,|\mathcal{U}_r^{\mathrm{tch}}|}\sum_{u \in \mathcal{U}_r^{\mathrm{tch}}} p_u \]
\[ \mathrm{PAS} \;=\; \frac{1}{|\mathcal{R}|}\sum_{r \in \mathcal{R}} \mathrm{PAS}_r \;\in\; [0,1] \]
LaTeX 源 
\mathrm{PAS}_r \;=\; \frac{1}{2\,|\mathcal{U}_r^{\mathrm{tch}}|}\sum_{u \in \mathcal{U}_r^{\mathrm{tch}}} p_u

\mathrm{PAS} \;=\; \frac{1}{|\mathcal{R}|}\sum_{r \in \mathcal{R}} \mathrm{PAS}_r
论文里也可报fraction-of-fully-aligned变体 \(\mathrm{PAS}^{\star}_r = \frac{1}{|\mathcal{U}_r^{\mathrm{tch}}|}\sum_u \mathbb{I}[p_u = 2]\) 作 strict 版（更接近 reviewer 直觉）。

{profile}

{utterance}

                  for e in exec_log if e.get("teacher_output")]

PQS — Pedagogical Quality Score aggregate of 3 KELE sub-metrics · headline · Q2.b · ∈ [0,1] · = mean(KELE NDAR + SPR + IAR)
是什么 / 评估目标
PQS 测"MAS 教学是否遵循好教学法"——3 个独立子轴：(i) NDAR — 教师不直接 reveal accepted answer 而是引导学生自己想；(ii) SPR — 教学有 scaffolding phase 结构（introduce → guide → consolidate）；(iii) IAR — 提问 vs 告知比例足够高（Socratic）。3 项算术平均。PQS不是 reviewer 容易质疑的"我们发明的"指标——3 个 sub 都是 KELE \citep{peng2025kele} 已经发表的命名指标。
参考工作
3 个子项直接 port 自 KELE \citep{peng2025kele} 的 4-pedagogical-metrics 中的 3 项（NDAR / SPR / IAR；KELE 的第 4 项 PRR 已划到 PAS 里）。Scaffolding 教学理论本身可追溯到 Vygotsky / Wood et al. 1976；Socratic 教学的 IAR 量化在 Aligning-Pedagogy \citep{dinucu2025aligningpedagogy} 的 helpfulness sub-rubric 中也有出现。TreeInstruct \citep{kargupta2024treeinstruct} 把 scaffolding 拆成"identify error → ask diagnostic question → guide fix"3 stage，与 SPR 的 introduce/guide/consolidate 是同源思路。
输入
(plan JSON, execution_log) — 需要 plan 的 step.type / step.phase 元数据来判 SPR；execution_log 的 teacher_output 用于判 NDAR / IAR。
计算公式
对 run \(r\)： \[ \begin{aligned} \mathrm{NDAR}_r &\;=\; \frac{1}{|\mathcal{U}_r^{\mathrm{tch}}|}\sum_{u \in \mathcal{U}_r^{\mathrm{tch}}} \mathbb{I}[\,\mathrm{reveal}_J(u, \alpha_r) = \mathrm{none}\,] \quad \text{(LLM judge)} \\ \mathrm{SPR}_r &\;=\; \frac{|\{\mathrm{phase} \in \{\mathrm{intro}, \mathrm{guide}, \mathrm{consol}\} : \mathrm{phase\_present}(r, \mathrm{phase})\}|}{3} \\ \mathrm{IAR}_r &\;=\; \min\!\left(1,\; \frac{\#\mathrm{questions}(r)}{\#\mathrm{statements}(r) + 1}\right) \end{aligned} \] 其中 \(\mathrm{reveal}_J \in \{\mathrm{full}, \mathrm{partial}, \mathrm{none}\}\) 由 judge 判 "是否在学生 attempt 之前 reveal accepted answer"；phase_present 由 rule-based 按 step.metadata.phase 或 LLM-tag 检测；questions / statements 计数用 regex (含 ? 或 imperative-question pattern)。聚合：
\[ \mathrm{PQS}_r \;=\; \frac{1}{3}\!\left(\, \mathrm{NDAR}_r + \mathrm{SPR}_r + \mathrm{IAR}_r \,\right), \qquad \mathrm{PQS} \;=\; \mathbb{E}_r[\mathrm{PQS}_r] \]
LaTeX 源 
\mathrm{NDAR}_r \;=\; \frac{1}{|\mathcal{U}_r^{\mathrm{tch}}|}\sum_{u} \mathbb{I}[\mathrm{reveal}_J(u, \alpha_r) = \mathrm{none}]

\mathrm{SPR}_r \;=\; \frac{|\{\mathrm{phase} \in \{\mathrm{intro}, \mathrm{guide}, \mathrm{consol}\} : \mathrm{present}\}|}{3}

\mathrm{IAR}_r \;=\; \min\!\left(1,\; \frac{\#\mathrm{questions}(r)}{\#\mathrm{statements}(r) + 1}\right)

\mathrm{PQS}_r \;=\; \frac{1}{3}\!\left(\mathrm{NDAR}_r + \mathrm{SPR}_r + \mathrm{IAR}_r\right)
论文里报 PQS 平均值 + 三子项 breakdown（appendix table），便于诊断"是哪一维度差"。

Accepted answer: {accepted}
Teacher utterance: {utterance}

def ndar_run(exec_log, accepted_answer, judge="gpt-5") -> float:
    utts = [e["actual_interaction"]["teacher_output"]
            for e in exec_log if e.get("teacher_output")]
    rev = [json.loads(call_llm(judge, NDAR_PROMPT.format(accepted=accepted_answer, utterance=u),
                                temperature=0.0))["reveal"] for u in utts]
    return sum(1 for r in rev if r == "none") / len(rev) if rev else 0.0

def spr_run(plan, exec_log) -> float:
    phases_present = set()
    for s in plan["output"]["steps"]:
        ph = s.get("metadata", {}).get("phase")
        if ph in ("intro", "introduce", "guide", "consolidate", "consol"):
            phases_present.add(_canonical(ph))
    return len(phases_present & {"intro", "guide", "consol"}) / 3

import re
def iar_run(exec_log) -> float:
    utts = [e["actual_interaction"]["teacher_output"]
            for e in exec_log if e.get("teacher_output")]
    text = " ".join(utts)
    q = len(re.findall(r"\?", text)) + len(re.findall(r"\b(what|why|how|can you|describe)\b", text.lower()))
    s = len(re.findall(r"[\.!]", text)) - q
    return min(1.0, q / max(s, 1))

def pqs_run(plan, exec_log, accepted_answer, judge="gpt-5") -> float:
    return (ndar_run(exec_log, accepted_answer, judge) +
            spr_run(plan, exec_log) +
            iar_run(exec_log)) / 3

rsol — Post-Dialog Solve Rate LLM judge for student final code · headline · Q2.c · ∈ [0,1] · directly ported from Aligning-Pedagogy
是什么 / 评估目标
rsol 测"plan 跑完后，学生 agent 真的学会了吗"——具体是学生 agent（固定 Llama-3.1-8B-Instruct）在最后一个 subtask 内独立输出的代码，是否与 accepted_answer 功能等价。这是 Tier 2 最 high-stakes 的指标——直接对应论文 abstract 里"PersonalPlan 能让学生真的学会"的论断；所有方法都为提升 rsol 服务。rsol 与 EVR 互补：EVR 测 plan 可执行性，rsol 测教学有效性——可能 EVR 高但 rsol 低（plan 跑通但学生没学会，因为 plan 内容不对症）或反之（plan 偶尔 crash 但学生在跑通的 run 内真学会了）。
参考工作
这直接 port from Aligning-Pedagogy \citep{dinucu2025aligningpedagogy} 的核心 reward \(r_{\text{sol}}\): "post-dialog solve rate"。他们的设定是 math tutoring，学生在对话后被 sample K=8 次最终答案对 gold 数学答案验证。我们 port 到编程教学：学生在 plan 执行完后的最终代码尝试，对 Stack Overflow accepted_answer 验证 functional equivalence (HumanEval pass@1 \citep{chen2021codex} 同款判定)。论文里建议直接命名为 "Post-Dialog Solve Rate (\(r_{\text{sol}}\))" + cite Aligning-Pedagogy + Chen 2021，不引入 LST 等新缩写。TreeInstruct \citep{kargupta2024treeinstruct} 的"学生最终 fix 与 ground-truth 同构"是 same idea 在 debug 场景的 instantiation。
输入
(plan JSON, execution_log, accepted_answer) — 从 execution_log 中提取最后一个 subtask 内学生 agent 的所有 student_response，抽出最长的 fenced code block。
计算公式
令 \(\mathcal{R}_{\mathrm{valid}} = \{r \in \mathcal{R} : \mathrm{pat}_r \ne \mathrm{perfect\_from\_start}\}\) 是排除话题盲区失败 run 后的有效集合（pattern 由现有 evaluator.py 的 IQ check 标记）；\(\mathrm{code}_r^{\mathrm{stud,final}}\) 是 run \(r\) 中最后一个 subtask 内学生 agent 输出的最长代码片段；\(\alpha_r\) 是该 run 的 accepted answer；\(\equiv_J\) 是 LLM judge \(J\) (GPT-5) 判定的功能等价（同 EVR 的 CCR）。
\[ r_{\mathrm{sol}} \;=\; \begin{cases} \dfrac{1}{|\mathcal{R}_{\mathrm{valid}}|}\displaystyle\sum_{r \in \mathcal{R}_{\mathrm{valid}}} \mathbb{I}\!\left[\, \mathrm{code}_r^{\mathrm{stud,final}} \,\equiv_J\, \alpha_r \,\right], & |\mathcal{R}_{\mathrm{valid}}| > 0,\\[6pt] \mathrm{undefined}, & \text{otherwise}. \end{cases} \]
LaTeX 源 
r_{\mathrm{sol}} \;=\;
\begin{cases}
\dfrac{1}{|\mathcal{R}_{\mathrm{valid}}|}\displaystyle\sum_{r \in \mathcal{R}_{\mathrm{valid}}} \mathbb{I}\!\left[\, \mathrm{code}_r^{\mathrm{stud,final}} \,\equiv_J\, \alpha_r \,\right], & |\mathcal{R}_{\mathrm{valid}}| > 0,\\
\mathrm{undefined}, & \text{otherwise}.

论文里同时报 \(|\mathcal{R}_{\mathrm{valid}}| / |\mathcal{R}|\)（有效 run 比例）防止读者误解分母。

def r_sol_run(plan, exec_log, judge="gpt-5"):
    # 若 IQ 标 perfect_from_start，返回 None（不计入分母）
    if _has_perfect_from_start_pattern(exec_log): return None
    last_subtask = plan["output"]["subtasks"][-1]["id"]
    last_resps = [e["actual_interaction"].get("student_response", "")
                  for e in exec_log
                  if e.get("subtask_id") == last_subtask and e.get("requires_human_input")]
    if not last_resps: return 0
    final_code = _extract_longest_code(last_resps)
    if not _compiles(final_code, plan["meta"]["language"]): return 0  # hard 约束
    return functional_eq(final_code,
                         plan["input"]["accepted_answer"],
                         plan["input"]["query"],
                         plan["input"]["accepted_answer"], judge)

def r_sol(runs, judge="gpt-5") -> float:
    scores = [r_sol_run(p, log, judge) for p, log in runs]
    valid = [s for s in scores if s is not None]
    return sum(valid) / len(valid) if valid else 0.0

↑ higher better。Q2.c headline，论文最 high-stakes 指标。SOTA ~0.30-0.45，PersonalPlan 目标 ≥ 0.55。直接对应 Aligning-Pedagogy 报告的 rsol 量级。

11.7 Tier 3 — 学习者偏好（2 项，Sati. 见上面 §11.2）

JCC — Krippendorff's \(\alpha\) (Judge Cross-Consistency) established (Krippendorff 1980/2004) · ∈ [-1,1]
是什么 / 评估目标
JCC 测的是"3 个跨家族的 LLM judge 在同一对 pairwise 上的标签一致性"——它不评 method，评的是评估器本身的可信度。如果 JCC = α = 0.3，说明 3 个 judge 严重分歧，那么 Sati. 这个分数本身就不可信，整个 Tier 3 的结论都要打折扣。JCC 是 Sati. 的透明度指标，必须在 appendix 里报。
参考工作
Krippendorff's α 是跨学科的标准 inter-rater reliability 指标，1980 年提出 \citep{krippendorff2004content}（也常引为 Krippendorff 2018 ISBN 1506395678）；同族指标 Cohen's \(\kappa\)（两个 rater）、ICC（连续 / 多 rater）。NLP 论文用 α 评 inter-annotator agreement 是惯例：KELE \citep{peng2025kele} 报 ICC 0.68-0.83，Personality-Sim \citep{liu2024personalitysim} 报 \(\kappa\) 0.7+。我们采用 α（nominal level）因为 Sati. 标签是 3 类 categorical。
输入
3 个 judge（GPT-5 / Claude Opus 4.6 / Gemini 3 Pro）在同一对 pairwise 上的 choice 标签（"A"/"B"/"Tie"）。
计算公式
令 \(M = 3\) 个 judge、\(N\) 对 pairwise；\(y^{(m)}_i \in \{\mathrm{cand}, \mathrm{gold}, \mathrm{tie}\}\) 是 judge \(m\) 对第 \(i\) 对的标签；\(\delta(\cdot, \cdot)\) 是 nominal-level distance（不同 label 为 1，相同为 0）。
\[ D_o \;=\; \frac{1}{N\binom{M}{2}}\sum_{i=1}^{N}\sum_{1 \le m_1 < m_2 \le M} \delta\!\left(y^{(m_1)}_i, y^{(m_2)}_i\right) \]
\[ D_e \;=\; \frac{1}{n(n-1)}\sum_{c \ne c'} n_c\, n_{c'}\, \delta(c, c'), \qquad n = NM, \quad n_c = \#\{(m, i) : y^{(m)}_i = c\} \]
\[ \mathrm{JCC} \;=\; \alpha \;=\; 1 - \frac{D_o}{D_e} \;\in\; [-1, 1] \]
LaTeX 源 
D_o \;=\; \frac{1}{N\binom{M}{2}}\sum_{i=1}^{N}\sum_{1 \le m_1 < m_2 \le M} \delta\!\left(y^{(m_1)}_i, y^{(m_2)}_i\right)

D_e \;=\; \frac{1}{n(n-1)}\sum_{c \ne c'} n_c\, n_{c'}\, \delta(c, c'), \qquad n = NM, \quad n_c = \#\{(m, i) : y^{(m)}_i = c\}

\mathrm{JCC} \;=\; \alpha \;=\; 1 - \frac{D_o}{D_e}
直接调 krippendorff.alpha(reliability_data, level_of_measurement="nominal") 即可。
参考代码
import krippendorff
import numpy as np

def jcc(pairwise_results: dict[str, list[str]]) -> float:
    # pairwise_results: {"gpt-5": [...], "claude": [...], "gemini": [...]}
    label_map = {"A": 0, "B": 1, "Tie": 2}
    matrix = np.array([[label_map[c] for c in pairwise_results[j]]
                       for j in pairwise_results])
    return krippendorff.alpha(reliability_data=matrix,
                              level_of_measurement="nominal")
✅ 通过示例（α 高）
100 对 pairwise：GPT-5 选 cand=72, gold=18, tie=10；Claude 选 cand=70, gold=20, tie=10；Gemini 选 cand=68, gold=22, tie=10。3 judge 之间大多数 case 一致 → α ≈ 0.72（acceptable 区间），Sati. 数据可信。
❌ 不通过示例（α 低）
100 对 pairwise：GPT-5 选 cand=80（很喜欢 candidate）；Claude 选 cand=30（很喜欢 gold）；Gemini 选 cand=55（中立）。3 judge 在过半 case 上不一致 → α ≈ 0.25。Sati. 数据不可信——可能是 rubric 模糊、prompt injection 没防干净、或 judge 真的对这个任务无共识。修复方案：检查 rubric anchor 是否清晰、加 100 条人工 calibration、考虑加第 4 个 judge。
方向 / 阈值
↑ higher better。α 是 Sati. 可信度的 proxy：
  • α ≥ 0.8：excellent agreement（罕见）
  • α ≥ 0.667：acceptable for tentative conclusions（Krippendorff 推荐阈值）
  • α ≥ 0.5：本论文最低接受阈值
  • α < 0.5：Sati. 数据不可信，需要重做 rubric 或加 calibration
论文里报在 appendix table；KELE 实测 ICC 0.68-0.83，我们预期 α 0.55-0.70。

论文方法明确定义了 4 项 GRPO reward：R_struct、R_pers、R_ped、R_hard。每项 reward 应该有对应的 evaluation 指标，否则 RQ3 ablation 没法支撑。下表给出对应：
