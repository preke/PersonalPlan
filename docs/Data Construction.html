A Personalized Planning Dataset for Programming Education: Construction Workflow and Dataset Statistics

Dataset construction report · v15 release · May 2026
Abstract. We describe the construction of a dataset of 3,043 personalized multi-agent teaching plans built from Stack Overflow duplicate-question clusters. Each plan is generated by Claude Sonnet 4.6 from a triple of (technical question, learner profile, accepted answer) and is verified by a two-stage check that combines a deterministic structural rule and an LLM execution audit. The release covers 1,730 distinct questions; 971 of those carry two or more learner profiles, giving the dataset a natural cross-profile axis for studying personalization. We iterated the generate-evaluate-rewrite loop 10 times before declaring the release final. This report documents the pipeline and reports descriptive statistics on the resulting dataset.

1. Motivation

Most teaching-plan datasets treat instruction as a one-size-fits-all script. The same question receives the same answer, regardless of who is asking. That assumption breaks the moment one tutors a real audience: a senior C++ engineer who hits a Python iterator bug needs a different explanation than a first-year student who has never seen a generator, even when the underlying fix is the same.

We set out to construct a dataset that captures this distinction. Each sample combines a real Stack Overflow question, a learner profile with stated skills and self-description, the accepted Stack Overflow answer, and a structured teaching plan that uses the profile to shape the explanation. Stack Overflow's duplicate-question feature gave us the lever we needed: many technically equivalent questions have been asked by users with very different backgrounds, so the same underlying problem is naturally associated with multiple learner contexts.

2. Pipeline at a Glance

Figure 1. Pipeline for quality-driven personalized teaching plan generation and validation. Four stages connect Stack Overflow data ingestion to the final personalized planning dataset: (1) data collection with duplicate-original grouping and basic cleaning; (2) LLM-based quality filtering on both questions and learner profiles; (3) plan generation against an explicit tool pool, producing a structured JSON plan; and (4) two-stage validation — a static structure check followed by an execution-effectiveness check whose outcome is judged by an LLM. Plans that fail either validation stage are returned to plan generation for revision; only plans that clear both stages enter the personalized planning dataset.

3. Data Collection and Clustering

Our crawler walks Stack Overflow looking for the original_questions field, which marks a question as a duplicate of another. Following each such link, we pull the original question and group it with its duplicates. We treat the cluster, rather than the single question, as the basic unit. Duplicates within a cluster carry exactly the variation we want: different ways real users phrase the same underlying problem, often from different backgrounds.

From a raw crawl of tens of thousands of question pages we retain only clusters whose original question has an accepted or strongly upvoted answer. After cleaning we keep three fields per sample: question (title, body, context), profile (the asker's self_description and skills), and answer (accepted answer, falling back to the highest-voted answer when no accepted one exists). Every other field is discarded. The cluster structure naturally yields a one-to-many shape: a single canonical question is paired with several learner profiles, each forming a separate sample.

4. Quality Filtering

Not every (question, profile, answer) triple is worth keeping. We apply an LLM filter that scores each candidate along two axes.

Question quality. We grade questions with the accepted answer in the prompt, so the model can ground its judgement on something concrete. We ask whether the answer in fact solves the question; whether the question demands stepwise reasoning rather than a one-line lookup; whether multiple solution paths are defensible, giving the plan room to teach trade-offs; and whether the topic naturally splits into stages a teacher would want to traverse (concept, common mistake, demonstration, practice, transfer). Roughly two thirds of candidate questions fail at least one of these criteria.

Profile quality. The rule is simpler: we keep the profile only if self_description says something concrete — a job title, a stack, an educational background, a stated interest. Profiles that are blank, full of greetings, or composed only of personal links are dropped. The reason is mechanical. If the profile carries no personalization signal, no downstream plan can adapt to it, and we end up generating generic teaching regardless of how good the question is.

After this stage, 1,733 question groups survive. Each group expands into a small set of (question, profile) samples by duplication. 3,062 such pairs entered plan generation; 3,043 cleared the two-stage validation and form the v15 release.

5. Plan Generation

The generator is Claude Sonnet 4.6. We feed it the technical question, the accepted answer, the learner profile, and a description of an external tool pool the plan is allowed to call. The model returns a single JSON object with three top-level fields — agents, subtasks, and execution_order — together with per-step tool assignments.

5.2 What the prompt asks the model to do

The prompt directs the model to extract the destination concept from the accepted answer rather than copying the answer verbatim; to read the profile and pick a personalization strategy (an analogy from the learner's stack, a level adjustment, a prerequisite scaffold); to decompose the problem into subtasks with observable outputs; to assign the minimal set of agents and give each only the tools it actually uses; and to keep depends_on links forward-pointing and execution_order coverage exact.

We chose strict JSON output over free-form text after an early prototype demonstrated that prose plans, while pleasant to read, were essentially unevaluable. With a schema in place we can count steps, audit tool calls, and detect dependency cycles automatically.

6. Two-Stage Validation

Generation alone does not guarantee quality. The model can produce plausible-looking JSON that does not in fact carry out the teaching task. We catch this by running every generated plan through two independent checks in series: a static structural check, followed by an execution-level check that actually runs the plan and has an LLM judge the outcome.

Stage A — Static structural check. A deterministic Python checker parses the JSON and runs roughly thirty assertions on its shape: is the JSON valid; does every referenced agent exist in the agents array; are all tool names drawn from the declared tool pool; is depends_on resolvable and acyclic; does execution_order cover every step exactly once; do the tools an agent actually uses match the tools the agent declares; if loops appear, do they have well-formed termination conditions. Failures are tagged by severity (severe / major / minor) and mapped to a bucket (Excellent / Good / Acceptable / Failing). The check is fast, reproducible, and entirely format-level — it cannot tell whether the plan teaches anything sensible, only whether it is shaped correctly.

Stage B — Execution validation. Plans that survive Stage A move to execution. The workflow defined by the plan is actually carried out: each step is dispatched to the agent that owns it, with the declared tool invoked, and the recorded outputs and intermediate states are collected. The execution trace is then handed to an LLM judge, which decides whether the execution achieved the plan's stated objective. The judge is prompted with the original question, the learner profile, the accepted answer, the plan itself, and the execution trace, and asked five concrete things: does the execution actually advance the problem toward a solution; are the tool calls reasonable in context; do the steps connect logically across the workflow; is the produced content aligned with the question and accepted answer; and does the personalization strategy genuinely reflect the learner profile rather than merely mention it. Stage B is where teaching-content errors surface — for example, a plan that is structurally perfect but executes into instructions that teach a different solution than the accepted answer recommends.

A plan enters the release only when Stage A returns Good or Excellent and Stage B confirms that the execution achieves the plan's objective. Plans that fail either stage are sent back to Stage 3 to be regenerated, and the regenerated plans re-enter Stage A and Stage B from the top.

7.5 Agent-role diversity

7.7 Empirical evidence of cross-profile personalization

To confirm that profile differences translate into observable plan differences, we drew a random sample of 10 questions, each with 2–3 profiles, and asked an independent LLM judge to compare the plans for the same question side by side. The judge graded each group on four points: profile diversity, plan content diversity, profile→plan alignment, and an overall verdict. Results:

VerdictGroupsMeaning
STRONG8Plans clearly tailored to each profile; swapping profiles would make analogies and recommendations nonsensical.
SURFACE2Plans share core content; differences are visible but largely cosmetic.
NONE / NON_TRACKING0No alignment between profile differences and plan differences.

Table 6. Cross-profile audit on 10 randomly drawn question groups covering 24 plans in total.

To make the verdict concrete: for the question "How do you create a REST client for Java?", the plan paired with a Java engineer (with the web-services skill) commits to the standard library and walks through HttpURLConnection method by method; the plan paired with a UX lead whose skills include apache-flex and flash reframes the same question as a library-landscape problem (HttpComponents, OkHttp, Retrofit) and uses a Flex URLLoader analogy in its first subtask. Similar profile-tied divergence appears in 7 of the other 9 groups.
