Baseline 实施细节

PersonalPlan v2 lineup（2026-05-19）的 8 个 active baseline。所有 baseline 都遵守同一个 contract：

输入 (query, learner_profile) 一对 → 输出 一个符合 §9 JSON schema 的多智能体 plan （顶层 {input, output}，output 含 agents / subtasks / execution_order）。

命名约定

代号类别含义
L1-L3Bare LLM闭源旗舰 LLM 单次裸调用，无任何 agent 协调
F1-F2Framework MAS领域通用多智能体框架（AutoGen / AutoAgents 直接给框架跑）
M1-M3Method-specific MAS专用 plan-generation 论文方法 port（AIPOM / AFlow / AOP）

⚠ 别跟论文的 Tier 1/2/3 混了——后者是评估协议的三层（静态 plan / MAS 执行 / preference 评判），跟这里的 baseline 类别代号无关。

共用基础设施（一处提，下面每个 baseline 就不重复了）

  • Backbone：L1=GPT-5、L2=Claude Opus 4.6、L3=Qwen3-Max；F* / M* 全用 Qwen3-32B（DashScope，OpenAI-兼容）
  • System prompt 前缀：所有 baseline 都用 baselines.common.prompt_sections.compose_t4() 拼装 PREAMBLE + §5 + §9 + §12（即任务陈述 + 工具池 + §9 schema + 结尾约束），约 4500 字符。§9 schema 原文 verbatim 来自 ../prompt_for_inference.txt 第 9 章节
  • Runner：baselines.common.runner.run_baseline(plan_fn, ...)，断点续跑 + JSON repair + §9 strict 校验（schema_validator.py）
  • 输出布局：evaluation_results/baselines/<name>/{plans.jsonl, failures.jsonl, progress.json, native_outputs.jsonl}
  • API key：.env 放仓库根，含 DASHSCOPE_API_KEY（L3 + F* + M*）+ OPENAI_PROXY_API_KEY/OPENAI_PROXY_BASE_URL（L1 / L2）

下面 8 个章节，每个 baseline 一个，结构完全一致：怎么做 / 输入 / 怎么运行 / 适配 / 运行效果。

L1GPT-5（裸 LLM 单次调用）

怎么做

把 compose_t4() 作为 system message、(query, learner) 作为 user message，单次 chat 调用，直接拿模型返回的字符串里的 JSON 当 §9 plan。没有 agent 编排、没有 retry。

输入（代码位置）

  • System prompt 组装：baselines/t1/plan.py（调用 build_t1_system_message()，等价于 compose_t4()）
  • User message："QUERY: {query}\nLEARNER: {json}"，在 plan.py 内拼
  • Backend 路由：baselines/common/llm_client.py 的 LLMClient(backend="gpt-5")，走 OPENAI_PROXY_BASE_URL

怎么运行

# Smoke（跑 1 条）
python -m baselines.t1.run --backend gpt-5 --limit 1

# 全量（3043 条，断点续跑）
python -m baselines.t1.run --backend gpt-5

做了哪些适配

无。这是 baseline 集合里最朴素的实现——目的是给"闭源旗舰 LLM 单次 prompt 能达到的上限"做基线。没引入 retry、没引入框架包装、没引入二次 prompt。

运行效果

  • 单条耗时：~30s
  • 预期 §9 合规率：高（GPT-5 zero-shot 跟 schema 很好）
  • 典型失败：偶发输出多余 markdown fence，由 json_repair.fix_json_format() 兜底
  • 产物：evaluation_results/baselines/t1_gpt-5/plans.jsonl

L2Claude Opus 4.6（裸 LLM 单次调用）

怎么做

跟 L1 完全一样的 contract，只换 backend。

输入（代码位置）

  • System prompt + user message 组装位置同 L1（baselines/t1/plan.py）
  • Backend：LLMClient(backend="claude-opus-4-6")，走 OPENAI_PROXY_* 的 Anthropic-兼容端点

怎么运行

python -m baselines.t1.run --backend claude-opus-4-6 --limit 1
python -m baselines.t1.run --backend claude-opus-4-6

做了哪些适配

无方法层改动。Model id 跟代理实际支持的 id 对齐——如果代理报 model 不存在，改 common/llm_client.py 里 self.model = "claude-opus-4-6" 为代理认的那个串。

运行效果

  • 单条耗时：~30s
  • §9 合规率：跟 L1 同档；偶发 JSON 修复需求略多于 L1
  • 产物：evaluation_results/baselines/t1_claude-opus-4-6/plans.jsonl

L3Qwen3-Max（裸 LLM 单次调用）

怎么做

L1/L2 的开源旗舰对照点：同样的 contract，backend 换 Qwen3-Max（阿里灵积），让 reviewer 看到"闭源旗舰 vs 开源旗舰"在相同 prompt 下的差距。

输入（代码位置）

  • System prompt + user message 同 L1
  • Backend：LLMClient(backend="qwen3-max")，走 DashScope OpenAI-兼容端点

怎么运行

python -m baselines.t1.run --backend qwen3-max --limit 1
python -m baselines.t1.run --backend qwen3-max

做了哪些适配

无方法层改动。注意 DASHSCOPE_API_KEY 必须在 .env 里。

运行效果

  • 单条耗时：~30s
  • §9 合规率：低于 L1/L2（Qwen3-Max 没专门 schema-follow 训练，偶发会塞 <think> 块或额外解释）
  • 产物：evaluation_results/baselines/t1_qwen3-max/plans.jsonl

F1AutoGen（领域通用 MAS 框架包装）

怎么做

用 Microsoft AutoGen 的 AssistantAgent 包装一层，system_message 跟 L* 完全一样，让 AutoGen 框架来做 chat round-trip。本质上是"通用 MAS 框架 + 1 agent + 1 round"——AutoGen 不会自动分解任务，所以对单 deliverable（一份 §9 plan）的最忠实实现就是单 agent。

输入（代码位置）

  • System message：baselines/autogen_qwen3/plan.py:147 调用 build_t4_system_message()（= compose_t4()）
  • User message："QUERY: {query}\nLEARNER: {json}"，在 _user_message() 拼
  • Model client：OpenAIChatCompletionClient（DashScope qwen3-32b），_make_autogen_client()
  • Agent 实例化：每条样本新建 AssistantAgent（避免 model_context 在 3043 条间跨样本污染）

怎么运行

python -m baselines.autogen_qwen3.run --limit 1
python -m baselines.autogen_qwen3.run

做了哪些适配

  • 框架简化为单 agent（"AutoGen 多 agent 扩展"会引入超出 baseline contract 的结构选择，跳过）
  • Backbone 从论文示例的 GPT-4 换成 qwen3-32b（v1 baseline design 统一选择）
  • fallback 路径：如果 autogen-agentchat 没装，_make_direct_plan_fn() 直接走 LLMClient.chat()，架构等价

运行效果

  • 单条耗时：~30s
  • §9 合规率：跟 L3 相近（同 backbone）
  • 产物：evaluation_results/baselines/autogen_qwen3/plans.jsonl

F2AutoAgents（动态 role 生成 MAS）

怎么做

跑论文原仓库代码 Link-AGI/AutoAgents：Manager 自动生成一组 agent + plan，配合 ObserverAgents / ObserverPlans 多轮 review 修正，最后落 §9 JSON。我们的 adapter 把它当黑盒跑（子进程，独立 Python 3.10 venv）。

输入（代码位置）

  • Adapter 入口：baselines/autoagents/plan.py（主 venv）
  • 子进程脚本：baselines/autoagents/upstream_runner.py（跑在 .venvs/autoagents_310/，Python 3.10 + 245 pinned packages from external/AutoAgents/requirements.txt）
  • Idea 拼装：_build_idea() 把 compose_t4() 整段 + hard-instruction "FINAL output must be §9 JSON" 一起喂给 Manager（scheme A，避免后处理 schema 翻译）
  • 上游 clone：external/AutoAgents/，可通过 AUTOAGENTS_REPO 环境变量 override

怎么运行

# 先建子 venv（一次性）
python3.10 -m venv .venvs/autoagents_310
source .venvs/autoagents_310/bin/activate
pip install -r external/AutoAgents/requirements.txt
deactivate

# Smoke + 全量（adapter 自动 spawn 子进程）
python -m baselines.autoagents.run --limit 1
python -m baselines.autoagents.run

做了哪些适配

  • Backbone 替换：LiteLLM 接 DashScope OpenAI-兼容端点跑 qwen3-32b（论文用 GPT-4）
  • Manager hardcode 3 轮（vs explorer 的 n_round=5，paper 内部不一致，我们按 3 跑）
  • Scheme A：Manager 被指示直出 §9 JSON，跳过原来的"输出 markdown 再后处理"路径——避免方法保真度被翻译 LLM 二次污染
  • 已知 leak：上游 create_roles.py 只认 SearchAndSummarize 一个工具，跟我们 8 工具池不重合 → 偶发 step.tool 不在池里，做 null 兜底

运行效果

  • 单条耗时：~11 min（子进程跑多轮 Observer，是 8 个 baseline 里最慢的）
  • §9 合规率：中等（Manager 偶发输出 plain text，触发 json_repair 兜底）
  • 产物：evaluation_results/baselines/autoagents/plans.jsonl + native_outputs.jsonl（上游 Manager 原始输出留底）

M1AIPOM（agent-aware DAG planner）

怎么做

port AIPOM 论文（Kim et al., EMNLP 2025 Demo）的 single-shot DAG planner：四条规划原则（显式 agent 分派 / 类型化 I/O 依赖 / 可检视 DAG 结构 / 无隐式协调）一字不差编进 system prompt，让 LLM 按 AIPOM 风格直出 §9。不跑论文里的 conversational + graph-editing refinement loop——那是 human-in-the-loop demo，batch 评估里没人在回路。

输入（代码位置）

  • System prompt：baselines/aipom/meta_prompt.py 的 NEW_META_PROMPT（compose_t4() 前缀 + AIPOM 四原则 + 输出约束）
  • plan 入口：baselines/aipom/plan.py:plan_fn(query, learner)
  • User message："User query: {query}\nLearner profile: {json}\n\nOutput the JSON plan now:"
  • 上游 prompt 已内联，无需 clone AIPOM 仓库

怎么运行

python -m baselines.aipom.run --limit 1
python -m baselines.aipom.run

做了哪些适配

  • 跳过 refinement loop（人在回路 demo 路径不在 batch 评估范围）
  • Backbone 换 qwen3-32b（论文 demo 用 GPT-4o）
  • 输出从 AIPOM-native (node list + edge list) 换成 §9 schema 直出——避免 post-hoc translator

运行效果

  • 单条耗时：~30s（待 smoke 实测确认）
  • §9 合规率：预期跟 AOP 同档（qwen3-32b + 强 schema 约束）
  • Reviewer 核验关键：每个 step.agent 是否能在 output.agents[*].agent_role 里逐字找到；depends_on 是否对应真实数据流；execution_order 是否拓扑合理
  • 产物：evaluation_results/baselines/aipom/plans.jsonl + native_outputs.jsonl（raw + 解析后的 plan）

M2AFlow（operator-composition planner）

怎么做

port AFlow 论文（Zhang et al., ICLR 2025）的 operator-composition seed step：operator 池（Generate / Review / Revise / Ensemble / Custom）+ 四条 design rule（operator decomposition / parallelization / iterative refinement / code-style structure）一字不差编进 system prompt。不跑论文里的 MCTS 搜索循环——MCTS 是 task-class 级优化（一个 workflow 应用到一整个 benchmark），per-query 跑既不经济也是误用。

输入（代码位置）

  • System prompt：baselines/aflow/meta_prompt.py 的 NEW_META_PROMPT（compose_t4() 前缀 + operator 池 + 四规则）
  • plan 入口：baselines/aflow/plan.py:plan_fn(query, learner)
  • User message："User query: {query}\nLearner profile: {json}\n\nOutput the workflow as JSON now:"
  • 上游 prompt 已内联，无需 clone MetaGPT/AFlow 仓库

怎么运行

python -m baselines.aflow.run --limit 1
python -m baselines.aflow.run

做了哪些适配

  • 跳过 MCTS（论文最核心机制，但 per-query 不适用；详细论证见 baselines/aflow/__init__.py）
  • Backbone 换 qwen3-32b（论文搜索 GPT-4o backed）
  • 输出从 Python workflow 代码换成 §9 schema 直出——Review/Revise 循环映射到 §9 loop block；多 candidate Ensemble 映射到多 producer 的 depends_on

运行效果

  • 单条耗时：~30s（待 smoke 实测确认）
  • §9 合规率：预期跟 M1 同档
  • Reviewer 核验关键：step.objective 是否反映 operator 角色（Generate / Review / Revise / Ensemble / Custom）；有 Review 步时是否有 loop block；Ensemble 时 depends_on 是否列全 producer
  • 产物：evaluation_results/baselines/aflow/plans.jsonl + native_outputs.jsonl

M3AOP（meta-agent + worker pool + replan loop）

怎么做

port AOP 论文（Li et al., ICLR 2025）：meta-agent 把 query 分解到 8 个 teaching worker（ConceptTutor / CodeValidator / DocsRetriever / WebResearcher / FileWriter / PaperSearcher / RagRetriever / DirectoryReader），三条 AOP 原则（solvability / completeness / non-redundancy）作内部规划纪律。然后用 LLM-as-judge 给每个 step 打 [0,1] 分，分数 < 0.5 的 step 触发 replan，最多 2 轮。

输入（代码位置）

  • Meta-agent prompt：baselines/aop/meta_prompt.py 的 NEW_META_PROMPT（compose_t4() + 三原则 + 8-worker 池）
  • Worker 能力描述：baselines/aop/teaching_agents_descs.py（8 个 worker 各自的 capability_card）
  • plan 入口：baselines/aop/plan.py:plan_fn(query, learner)
  • Judge prompt：baselines/aop/plan.py 的 _JUDGE_PROMPT
  • 上游 patch：baselines/aop/aop_patch.py 处理上游 hard-coded 路径；clone 在 external/Agent-Oriented-Planning/

怎么运行

# 先 clone 上游
cd external && git clone https://github.com/lalaliat/Agent-Oriented-Planning.git && cd ..

python -m baselines.aop.run --limit 1
python -m baselines.aop.run

做了哪些适配

  • Worker 池从论文原 4 个（code/math/search/commonsense）扩成 8 个 teaching worker，与我们 §1 工具池一一对应
  • Reward 信号从原 SimilarityMLP 换成 LLM-as-judge（论文 MLP_high.pt 未公开，最大方法偏离，需诚实标注）
  • Batch replan：一次 LLM 调用修正整个 weak step 集合（论文是 single subtask per call，节省调用）
  • 没 port planner.py:96-189 的 plan_in_detail 分支

运行效果

  • 单条耗时：~40s（含 judge 调用 + 偶发 replan）
  • §9 合规率：高（最早就跑通的 baseline，schema 容错最稳）
  • 触发 replan 时 stdout 打 [aop] LLM-as-judge replanning loop fired for this plan.
  • Reviewer 核验关键：step.agent 必须是 8 个 worker 里的一个（roster 写死）；tool 字段跟 worker capability 对齐
  • 产物：evaluation_results/baselines/aop/plans.jsonl + native_outputs.jsonl（initial plan + replan rounds + final plan）
