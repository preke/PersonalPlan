1.2 七条硬性约束

  1. 统一 backbone：所有 trainable / same-backbone reference 用 Qwen-32B-Instruct（保 v1 同尺寸 backbone 候选见 §8.2）。
  2. 统一 plan schema：所有 baseline 输出同一 JSON（agents / subtasks / steps / execution_order）。
  3. 统一推理 prompt：用 GP 在 5/14 完成的 inference prompt（4 处最小改动版）。
  4. 判卷者跨模型族：GPT-5 主判 + Claude Opus 4.6 复核，order randomization + criterion-by-criterion。
  5. 按 question_id 切分：Test ∩ Train = Test ∩ Dev = Dev ∩ Train = ∅；切分键是 question_id，不是行号（防止同 query 不同 profile 形成泄漏）。
  6. SFT 与 GRPO 共享 Train（10 篇近期 paper 中 9 篇这么做）；Dev 仅作 ckpt selection 与 gate；Test 全程封存。
  7. GRPO 训练期不调 LLM judge：所有 reward 都是规则式 O(|P|) Python；LLM judge 仅在最终 Test 评估时调用。

具体实现脚本（已落地为 build_splits.py，跑一次冻结结果）：

import json, random
from collections import defaultdict

SEED = 42
qid_to_profs = defaultdict(set)
with open("multi_agent_dataset_filtered_qap.jsonl") as f:
    for line in f:
        d = json.loads(line)
        qid_to_profs[str(d["question_id"])].add(int(d["profile_index"]))

strata = defaultdict(list)
for q, profs in qid_to_profs.items():
    strata[len(profs)].append(q)

rng = random.Random(SEED)                       # 独立 RNG 实例，避免被外部 random state 干扰
train_q, dev_q, test_q = [], [], []
for k in sorted(strata.keys()):                 # 排序遍历桶，保证跨机器 deterministic
    ql = strata[k]
    rng.shuffle(ql)
    n = len(ql); n_test = round(n*0.10); n_dev = round(n*0.10)
    test_q  += ql[:n_test]
    dev_q   += ql[n_test : n_test + n_dev]
    train_q += ql[n_test + n_dev:]

json.dump({"train_qids": train_q, "dev_qids": dev_q, "test_qids": test_q},
          open("splits/maple_split_v1.json", "w"), indent=2)
# 行级 split 由 qid 集合派生，禁止直接切行
# 派生后写入：SFT/data/{pad,sdp}/{train,dev,test}.jsonl 与 GRPO/data/grpo/{train,dev,test}.jsonl

§2.3 符号说明

  • (query, profile) = 输入元组，分别对应 plan.input.query 和 plan.input.learner（含 about_me + top_tags）。
  • (agents★, subtasks★, steps★, execution_order★) = gold plan 的四个组成部分；按论文方法分成 scaffold（agents+subtasks，由 PAD 学）和 execution（steps+execution_order，由 SDP 学）。

Dev 监控（不 early stop，只选 ckpt）

  • 每 50 step 算一次 Dev NLL（仅 target tokens：PAD 算 agents+subtasks tokens；SDP 算 steps+execution_order tokens）。3 ep = ~9 个 eval point，2 ep = ~6 个。
  • 每 epoch 末额外采样 50 条 Dev 跑生成，算 SV（schema validity）和 AR（acyclicity rate）。两项是廉价的，每次几分钟。
  • Checkpoint 选择：训完所有 epoch，挑 min(dev_NLL) 且 dev_SV ≥ 0.85 的 ckpt 作为该 Stage 出口。
  • Early stop 仅在发散：dev NLL 连续 2 次升 > 10% 或出现 NaN → 手动停。否则训完计划 epoch。

(4) SFT 阶段特有注意事项

  1. LoRA adapter 保存陷阱（ben 5/14 实测）：autotrain / peft 默认会自动 resize tokenizer，导致 adapter 文件里包含完整 token embedding（几十 MB → 1.9 GB）。加载时 base model 不知道要 resize → 报形状不匹配。解决：在加载脚本里检测 len(adapter_tokenizer) 是否等于 base.get_input_embeddings().weight.shape[0]，不一致就 resize。
  2. SDP 训练用 gold scaffold（不是 PAD 输出）：这是 exposure bias 的来源，但论文 §4 明确这么做，由 Stage 2 Joint Alignment 修。不要在 Stage 1b 阶段就改成喂 PAD 输出——会破坏可比性。
  3. Dev NLL 单降 ≠ 生成质量升：模型可能学会预测"高频 token 序列"但生成时仍然崩溃。所以 SV/AR 廉价抽测是 NLL 的必要补充，不能省。
  4. 不要在 SFT 期间调 LLM judge：(a) 慢（一次 100 条要 5–10 分钟），(b) 贵（每 50 step 算就是 ~100 个 eval point × 100 条 × 2 judge = 20,000 调用），(c) 训练循环和最终评估共用 judge 会形成弱泄漏（虽然在 dev 上）。LLM judge 只在 Stage 2 末跑一次 sanity，正式 judge 留给 Test。
  5. Stage 2 Phase A 的 reward 怎么算：直接对采样到的 scaffold 与 gold scaffold 算结构相似度 (agents multiset Jaccard + subtask name Jaccard，等权)，不调 SDP、不调 LLM judge。两点理由：(a) Phase A 的论文动机是修 PAD↔SDP coordination bias，结构信号最贴这个动机；(b) 把 R_pers / R_ped 留给 GRPO 独享，消融 RQ3 时能干净分离"Phase A 贡献 vs GRPO 贡献"，避免同一 reward 在 DPO 和 GRPO 里被优化两遍。R_pers / R_ped 仍在 §3.3 GRPO 阶段全套使用。
  6. 梯度检查点必开：32B + bf16 + LoRA 不开 grad checkpoint 直接 OOM。开了之后步速降 ~30%，但能装下。

3.2 Stage 2 — Joint Alignment（K=1，修 exposure bias）

对应 4-Methodology.tex:112–139。一个 outer iteration 内交替两个 phase：

Phase动作损失
Phase A（修 PAD） 当前 PAD 采 2 个 scaffold → 用 scaffold-vs-gold R_struct（agents 多重集 Jaccard ½ + subtask 名称集 Jaccard ½）打分 → 挑 chosen/rejected scaffold。不调 SDP，纯 rule-based、O(\|scaffold\|)。 DPO on PAD：L = -E[log σ(β·Δ_PAD)], β=0.1, lr 5e-5
Phase B（修 SDP） 用 PAD（Phase A 更新版）采样 scaffold 作 input NLL on-policy SFT, 1 ep, lr 5e-5

预算：≤ Stage 1 的 20%（论文承诺），即 ≤ 7 h；不做 early stop，跑完即用。

§3.2 符号说明（DPO 公式 L = -E[log σ(β·Δ_PAD)]）
  • L = 损失函数（要最小化）。
  • E[·] = 在训练样本上取期望（实操就是 mini-batch 均值）。
  • σ(x) = 1/(1+e^(-x)) = sigmoid 函数，把任意实数压到 (0,1)。
  • β = 0.1 = DPO 温度系数；β 越大对偏好差距越敏感（论文用 0.1，主流值 0.1–0.5）。
  • Δ_PAD = Δ_PAD^{w,l} = chosen scaffold (w) 与 rejected scaffold (l) 之间的 log-likelihood 差。具体定义见论文 eq. 在 4-Methodology.tex:130–137：
      Δ_PAD^{w,l} = log[π_PAD(w|q,p) / π_ref(w|q,p)] − log[π_PAD(l|q,p) / π_ref(l|q,p)]
  • w / l = winner / loser 两个 scaffold。判定方式：直接对每个 scaffold 算 scaffold-level R_struct = 0.5·Jaccard(agent_roles) + 0.5·Jaccard(subtask_names)（vs gold scaffold），高者为 w，低者为 l。相等则丢弃这对样本。
  • π_ref（DPO 里）= 上一轮 PAD checkpoint（不是 GRPO 的 π_SFT）；每个 outer iteration 开始时刷新。
ben 5/14 已踩过的坑：DPO 数据若用 autotrain，chosen/rejected 必须是 Python list 字符串而非 JSON 文本；delta_ll 字段不能有 Infinity；adapter 含 embedding 时需要在加载脚本里自动 resize base model token embedding。详见 5_14/ben/問題.md。

3.4 ★ 诊断 Gate：pass@1 vs pass@8 on Dev

动机：GRPO 是"对同一 prompt 采样多个、用 reward 把好的概率推高"。如果 SFT 后模型采样多次也几乎不出好答案，GRPO 救不了。这个 gate 是在烧 5–14 天 GRPO 之前的强制 go/no-go。

该方法来自 grpo_session_summary.html §7–§8（pass@n 诊断），是 2025 RL 训练前的标准 sanity check。

(1) "pass" 的可计算定义

对一个生成的 plan P，定义：

pass(P) =  SV(P) == 1          # JSON schema 合法
       AND has_cycle(P) == False   # DAG
       AND DC(P) >= 0.5           # 至少 50% 依赖指向存在的 step
       AND ATR(P) >= 0.5          # agent-tool 至少半数合理

全规则式、O(|P|) Python、无 LLM judge。

(2) 采样协议

指标采样设置
pass@1从 Dev 抽 100 条 prompt，temperature=0，每条采 1 个 plan
pass@8同一批 100 条 prompt，temperature=0.9 / top_p=0.95，每条采 8 个 plan

定义：pass@1 = (#prompts with pass(P_1)=True) / 100；pass@8 = (#prompts with at least one pass(P_i)=True out of 8) / 100。

(3) 判定阈值

场景判定动作
pass@1 < 0.30 且 pass@8 < 0.40SFT 没学到不进 GRPO。回头修：数据质量？LoRA r 太小？epoch 不够？换更强 backbone？
pass@1 ≥ 0.30，pass@8 / pass@1 ≤ 1.2SFT 已饱和采样多样性太低，GRPO 难有空间。考虑：提高 temperature、用更弱的 SFT（减少 epoch）、或直接验收当前 π_SFT 不跑 GRPO
pass@1 ≥ 0.30，pass@8 / pass@1 ≥ 1.5适合 GRPO启动 Stage 3；GRPO 的工作就是把 pass@8 里的好 plan 推成 pass@1

§3.3 GRPO 符号说明（超参表）
  • π_θ = 当前在训的策略模型；θ 是可训参数（这里 = LoRA 权重）。GRPO 训练过程 = 不断更新 θ 让 reward 期望升高。
  • π_SFT = SFT 阶段结束的策略（Stage 2 出口）；GRPO 既用它作初始化，又把它作为 KL 锚点（参考 anchor），见下条。
  • π_ref = KL 锚点（reference policy）。我们把它固定为 π_SFT 不动（pinned），论文 §4 锁定这种做法。每一步训练计算 KL(π_θ ‖ π_ref)，让 π_θ 不会跑得离 π_SFT 太远（防止把 SFT 阶段学到的 personalization / pedagogy 遗忘）。
  • G = num_generations，单个 prompt 在 GRPO 阶段采样多少个 plan。论文 G=8。group-relative advantage 就是在这 G 个样本内做 z-score。
  • β（这里指 GRPO 的 KL 系数，不是 DPO 的 β）= KL 项在 loss 里的权重。L = - clipped_surrogate + β·KL。β 越大 π_θ 越被钉死在 π_SFT 附近；β=0 就是完全放飞。
  • ε = PPO 风格的 importance-ratio clip 阈值；标准值 0.2，限制单步策略更新幅度。具体见论文 eq. 在 4-Methodology.tex:207–214。
  • 1 step / 1 update = 处理 effective_batch_size 个 prompts × G 个 generations 的一次梯度更新。我们 effective_batch=8，G=8 → 一次 update 处理 64 个 plan。
  • 1 epoch = Train 全部 prompts 都被采样一次（≈ 2,433 / 8 ≈ 304 次 update）。

Reward 函数（4 项，全规则式）

R_struct = 0.25·DAG + 0.25·DC + 0.20·ATR + 0.30·StructSim(plan, gold★)
R_pers   = min( |grounded_steps| / |steps|, 0.8 )
R_ped    = 0.5·SubPrec(plan) + 0.5·PhaseCov(plan)
R_hard   = -10.0 · I[ schema_invalid ∨ has_cycle ∨ invalid_tool_call ]

R_total  = w_s·z̄(R_struct) + w_p·z̄(R_pers) + w_e·z̄(R_ped) + R_hard
默认 (w_s : w_p : w_e) = (0.55 : 0.25 : 0.20)
§3.3 Reward 符号说明

四项 reward（每个采样的 plan 算一个标量）

  • R_struct ∈ [0,1] — 结构性 reward。四个子项加权（权重见公式）：
    • DAG ∈ {0,1}：plan 的依赖图是否无环（拓扑排序通过 = 1）
    • DC ∈ [0,1]：Dependency Completeness = |D_satisfied| / |D_total|，依赖指向存在 step 的比例
    • ATR ∈ [0,1]：Agent-Tool Relevance（详见 §5.2 五项指标）
    • StructSim(plan, gold★) ∈ [0,1]：plan 与该 query 的 gold plan 在 5 个 signature（agent roles / subtask names / step ids / dependency edges / tool calls）上的平均 multiset Jaccard 相似度（论文 eq. 28）
  • R_pers ∈ [0, 0.8] — 个性化 reward。统计有多少 step 的 instruction 同时含 (1) 第二人称代词 (you/your/the learner) 且 (2) profile 的关键词（来自 learner.about_me 或 learner.top_tags），上限 0.8 防止 collapse 到单一 persona template（论文 eq. 30）。
    |grounded_steps| = 满足条件的 step 数，|steps| = 总 step 数。
  • R_ped ∈ [0,1] — 教学性 reward。两个子项各占一半：
    • SubPrec(plan) = subtask precedence rate，plan 里 ordered subtask pair 在"gold plans 离线挖出的先后关系集合 Π"里的命中比例
    • PhaseCov(plan) = phase coverage rate，6 个标准教学阶段（probe / retrieve-demonstrate / apply / validate / feedback / consolidate）的覆盖率（5 个二元 indicator 的均值，论文 §4 eq. 32）
  • R_hard ∈ {-10, 0} — 硬约束 闸门（disqualifying gate）。任何一条违反 → −10：
    • schema_invalid：JSON 解析失败或缺顶层 key
    • has_cycle：依赖图存在环
    • invalid_tool_call：step 调用了不在该 agent tools 列表里的工具
    I[·] = indicator function（条件为真返回 1，否则 0）；η = 10.0 = 违规惩罚幅度（论文锁定）。∨ = 逻辑 OR。

加权聚合（每个采样独立算一个 R_total，G=8 个组成 advantage 计算的 group）

  • z̄(R_x) — z-score 归一化，在 同一 prompt 的 G=8 个采样内做：z̄(R_x_i) = (R_x_i − μ_x) / (σ_x + ε_safe)，其中 μ_x / σ_x 是 G 个采样的 mean / std，ε_safe ≈ 1e-8 防除零。
    用意：消除三项 reward 的量纲差异（R_struct 在 [0,1]、R_pers 在 [0,0.8]、R_ped 在 [0,1]），让权重 w_s/w_p/w_e 真正反映"相对重要性"而不是被量纲混淆。
  • w_s, w_p, w_e = 三项 soft reward 的权重；默认 0.55 : 0.25 : 0.20（论文锁定），sum=1。RQ4 Pareto 扫的就是这三个权重（详见 §3.5）。
  • R_hard 不做 z-score，直接以 −10 加进 R_total——目的就是让硬违规压倒性地把 advantage 压成负值，而不是被 group 内 normalize 掉。
  • R_total 不是给单个采样独立打分的指标，它是要丢进 GRPO advantage 公式的：A_i = (R_total_i − mean(R_total)) / (std(R_total) + ε)，A_i 才是真正算梯度的"advantage"信号。
Dev composite 公式里为什么用 "raw"（不 z-score）？
监控用的 Dev_composite = 0.55·R_struct_raw + 0.25·R_pers_raw + 0.20·R_ped_raw + (1 − HardViolRate) 用的是原始值（_raw 后缀），不 z-score。
原因：z-score 是 同一 prompt 同一 group 内的相对量，跨 checkpoint 没意义；监控要在 不同时间点对比 dev 表现，必须用绝对值。HardViolRate = Dev 100 条里至少一项硬违规的比例。

Dev-based checkpoint selection（GRPO 关键）

为什么 GRPO 必须用 dev 选 ckpt：grpo_session_summary §4 + §10 + DAPO 论文都明确指出—— training reward 与 dev metric 经常不相关。GRPO 训练 reward 一直涨可能是 reward hacking（输出更长、更多 keyword stuffing），dev composite 上的真实质量可能在某点之后就开始下降。所以选 ckpt 的依据必须是 dev，不是 training reward。

训练异常停机条件（不是 ckpt 选择，是救火）

  1. KL 失控：KL(π_θ ‖ π_SFT) > 10·β = 0.4，连续 3 次 eval → 停机查 lr
  2. NaN / Inf：reward 或 loss 出现 NaN → 立即停
  3. Reward hacking 早期信号：training_reward 升 ≥ 50% 但 dev_composite 跌 ≥ 10% 持续 3 次 eval → 停机查 reward 函数
  4. Mode collapse：reward_std 连续 3 次 eval 跌 > 70% 且 entropy 跌 > 50% → 停机查 temperature / KL

这四条都是safety net，正常训练下不会触发；触发了通常是 bug 不是模型不行。

(5) Reward Hacking 检测信号（必须每天人工看一次）

这是 GRPO 特有的危险——training reward 一直涨但 dev 不动甚至跌。grpo_session_summary §4 给的 proxy-vs-gold 曲线判定是黄金标准：

信号含义动作

(6) GRPO 阶段特有注意事项

  1. 不要在训练循环里调 LLM judge：(a) 100 条 judge call 一次几分钟，每 30 step 一次就是 28 小时纯 judge call/14 天训练；(b) judge 自身有 verbosity / position bias，把它喂回训练 = 双重 hack 面（grpo_session_summary §3）。所有 reward 全规则式 O(|P|) Python，论文 §4 明确这么要求。
  2. Loss mask（重要）：GRPO loss 只在 模型生成的 plan token 上算梯度，不在 prompt token 上。如果用 TRL，GRPOTrainer 默认正确；如果用 verl / 自实现，确认 mask 是 completion_mask 而不是 response_mask（参考 WebAgent-R1 与 Aligning-Pedagogy 都强调这点）。
  3. KL anchor 一定要固定到 π_SFT：论文 §4 明确要求。不要用 TRL 默认的 "ref model 每 N 步刷新"——那是 PPO 风格不是论文要的。Aligning-Pedagogy 用 moving anchor，但论文坚持 pinned。
  4. training reward 不代表模型变好：DAPO 论文里专门讲过 training reward 和 validation accuracy 可以"相关性接近 0"。所以必须看 dev composite 而不是只看 train reward。这是 grpo_session_summary §10 的核心警告。
  5. Dev composite 用 raw 不用 z-score：训练用 z-score 是为了同 prompt 内 G=8 个 sample 比较；dev 监控是要在不同时间点之间比较，必须用绝对值（raw），否则不同 ckpt 不可比。
  6. vLLM 加速 rollout 谨慎：SHC 5/14 实测 vllm 0.20.2 与 torch 2.12 ABI 不兼容，直接关 vllm 走 transformers generate() 慢但稳。如果之后换 verl 框架可以重启 vllm；但先确保稳定再上加速。
  7. 断点恢复必须包含 RNG state：14 天训练中间挂了，恢复时要 load 完整 optimizer state + RNG state，否则采样序列变了 = 实际上是新训练。torch.save({"model": ..., "optimizer": ..., "rng_state": torch.get_rng_state(), "cuda_rng_state": torch.cuda.get_rng_state_all()})。
  8. Reward 权重一改就要重训：RQ4 的 5 个 Pareto run 都是 continue-train 0.3 ep 从主 π_θ 接着训，不是从 SFT 重头训。这样 5 个 run 共享同一起点，对比才公平。

Rubric 给 judge 的 instruction：

Judge from the learner's perspective on these four criteria (in this order of priority):
1. Skill Match: Which plan starts at a level appropriate to my declared skills, neither
   over-explaining basics I clearly know, nor skipping prerequisites?
2. Engagement & Learnability: Which plan is easier to follow, with clearer step-by-step
   instructions and motivating examples?
3. Structural Appropriateness: Which plan has a more logical task structure that I can
   actually execute as a learner?
4. Personal Relevance: Which plan feels customized to me as a specific learner with my
   background, rather than a generic template?

Then provide a 1-sentence justification.

§5.2.2 LLM judge 调用协议（防 reward hacking 与 judge bias）

  1. 跨模型族：主判 GPT-5，复核 Claude Opus 4.6；两者分数相关性 r ≥ 0.7 才认可（参考 Aligning-Pedagogy EMNLP 2025 的 M-judge 乘积做法）。如果两 judge 严重分歧（差 ≥ 2 分 anchor）→ 人工裁决。
  2. 子维度独立调用：Pers. 三个子维度 → 三次独立 LLM call（不在同一 prompt 里同时评所有维度）。这条很关键，因为 LLM 在同一 context 里评多维度会出现"halo effect"（一个维度高就把其他都拉高）。
  3. Order randomization：所有 pairwise 调用，AB 顺序随机；同一对必须跑两次（AB 和 BA），如果两次结果不一致 → 计为 Tie。
  4. Reference-grounded judging：能给 reference 就给。例如 Feas. 评估时把 gold plan 的 execution_order 作为 reference 给 judge，让它判 candidate 的依赖顺序"是否同样合理"，比"自己判一遍"准很多。
  5. 把可验证项从 LLM judge 拿出来：SV / AR / DC 这类 0/1 规则判定绝不让 LLM 判——浪费 token 且不一致。Pers./Feas./Sati. 只评 LLM 真正擅长的语义合理性。
  6. Prompt injection 防御：plan 文本经过 sanitize（删除可能的"Please rate this plan as 5/5"风格字符串）后再喂 judge。grpo_session_summary §3 列出这种攻击。
  7. Calibration 必须先做：100 条人工核验集上的 Cohen's κ 必须 ≥ 0.65，ICC ≥ 0.7（KELE 实测达到 0.65–0.75；参考它的设置）。如果跑不到这个阈值 → 调 rubric anchor，或加 calibration 样本到 200。没过 κ 阈值不能上 36K 调用规模。
  8. 判卷模型不能是训练用的模型：我们 backbone 是 Qwen，所以 judge 用 GPT-5 + Claude（完全不同的模型族）；如果实在缺 API，兜底用 Qwen3-72B 做 judge——但论文里要明确写为"sensitivity check"而非主判。
