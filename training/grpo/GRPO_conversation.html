USER 你给我讲一讲论文中为什么要叫做 Verifiable and Counterfactual Rewards；GRPO 设计原则是怎样的，简单易懂的话来讲

一、为什么叫 "Verifiable and Counterfactual Rewards"

名字直接对应了这个奖励体系最关键的两个设计哲学，每一个都在对抗一种典型的"奖励作弊"问题。

1. Verifiable（可验证的）——对抗"神经网络 judge 容易被骗"

如果用一个 LLM 当裁判去打分（"这个 plan 写得好不好？"），模型很快会学会讨好裁判而不是真正变好：写更长、塞专业词、模仿裁判的口味——这就是 reward hacking。

所以论文里结构正确性（$R^{\text{struct}}$）和先修关系（$R^{\text{ped,hard}}$）这两部分完全用规则算，不让神经网络掺和：

  • 是不是 DAG？拓扑排序一下就知道
  • 依赖完整不完整？数边就行
  • 工具调用合不合法？查 schema
  • 概念顺序对不对？比对预先建好的先修图 $K$

规则不会被花言巧语骗——这就是 "verifiable" 的含义：奖励信号有一个确定的、可机器复现的判定标准，模型唯一的提分路径就是真的把结构做对。

2. Counterfactual（反事实的）——对抗"个性化作弊"

个性化最容易作弊：模型在 plan 里塞一句 "considering you are a beginner..." 就能骗过一个朴素的判分器，但 plan 的结构本身根本没变。

论文的做法非常聪明——同一个 query，分别用真 profile $p$ 和随机换过的假 profile $\tilde p$ 各生成一个 plan，然后要求：

$$R^{\text{pers}} = \text{真profile的plan和真ground truth的相似度} - \text{真profile和假profile两个plan的相似度}$$

这里的关键是第二项：如果模型只是表面贴关键词、结构没变，那它换了 profile 也会生成几乎一样的 plan，这一项就接近 0，奖励上不去。只有真的让 agent 组成、步骤粒度随 profile 改变，才能拿到分。这就叫"反事实"——不是问"它好不好"，而是问"换一个 profile 它会不一样吗？"

所以这个名字 = 两个抗作弊机制的合称：
  • Verifiable 守住结构和先修这种客观的东西（用规则）
  • Counterfactual 守住个性化这种主观的东西（用对照实验）

二、GRPO 的设计原则（大白话版）

GRPO（Group Relative Policy Optimization）来自 DeepSeekMath，论文里在它基础上加了几个针对本任务的改造。一共五个核心设计：

原则 1：组内比较代替学一个 critic

传统 PPO 要训练一个 value model 估计"基线分数"，又贵又难收敛。

GRPO 的做法：对同一个 prompt 采 $G$ 条轨迹（论文里就是 $G$ 个 plan），直接用这 $G$ 个分数的 mean 和 std 做归一化：

$$A_i = \frac{R_i - \mu_R}{\sigma_R}$$

直觉：不需要绝对分数，只需要回答"这次采样比同一题的其他采样好还是差"。简单、便宜、稳定。

原则 2：KL 锚点钉死在 SFT 模型上（论文的关键改造）

普通 GRPO 的 KL 项是和"旧策略 $\pi_{\theta_{\text{old}}}$"比——这是个移动的目标，时间久了 RL 阶段会慢慢忘掉 SFT 学到的东西。

论文改成：

$$\beta \cdot \mathbb{D}_{\text{KL}}[\pi_\theta \,\|\, \pi_{\text{SFT}}]$$

也就是始终拿 SFT 后的检查点当参照系。

直觉：SFT 已经把个性化和教学法装进模型了，RL 只是来补"结构正确"的短板。所以要给 RL 戴个"安全绳"——可以往前走，但不许跑远到忘了之前学的东西。

原则 3：奖励分段路由（Segment-wise Credit）

一个 plan 里有 agent 段、subtask 段、step 段，它们对应的奖励完全不同——结构奖励的方差很大，如果把它平摊到所有 token，会把 agent token 的梯度淹没掉。

论文的做法是奖励按 token 所属段路由：

  • agent token → 拿 $R^{\text{pers}}$ 的 advantage
  • subtask token → 拿 $R^{\text{ped}}$ 的 advantage
  • step / dependency token → 拿 $R^{\text{struct}}$ 的 advantage

直觉：谁的活算谁的账。负责个性化的那部分 token 就用个性化奖励来推；不要让方差最大的结构奖励淹没所有梯度。

原则 4：硬门 + 软分（Hard Gate + Soft Reward）

普通做法是所有违规都扣点分；论文区分两类：

  • 严重违规（schema 错、有环、工具不存在）→ 硬扣 $-\eta$，相当于直接判负，不给"用别的优点凑回来"的机会
  • 软指标（结构相似度、教学法）→ 正常打分相加

直觉：违章是"一票否决"的事，不是"扣两分但其他地方加三分还能过"的事。这避免模型用"个性化加分"换"结构上的 bug"。

原则 5：反作弊护栏（Anti-Hacking Safeguards）

哪怕做了上面这些，模型还是可能找漏洞。论文加了四道护栏：

  1. Judge 随机化——每个 batch 换 judge 模型和 prompt，模型没法稳定迎合某一个裁判
  2. 对抗轨迹注入——10% 的样本是故意写得很烂（超长、塞关键词、假引用），要求奖励必须 ≤ 0
  3. Proxy-vs-Gold 监控——画一条"训练奖励 vs. 人工抽检"的曲线，一旦训练奖励还在涨但人工分数停了，就早停（这是 reward hacking 的典型信号）
  4. 自适应 $\beta$——一旦发现策略离 SFT 飘太远，就自动把 KL 系数调大拽回来

一句话总结

奖励的命名 = 抗作弊宣言：能用规则就用规则（verifiable），不能用规则的就用对照实验（counterfactual），尽量不让 LLM judge 当主裁。

GRPO 的设计 = 一句口诀："组内比较算 advantage，SFT 当锚不让飘，奖励分段不串味，违规硬扣不商量，反作弊护栏全程开。"

USER 你解释一下本篇文章的 GRPO 的设计部分

论文 GRPO 设计的完整解读

整个 GRPO 部分（Section 4.2）可以理解成一句话：

"用一个抗作弊的奖励 + 一个钉死在 SFT 的策略优化算法，把'结构正确性'内化进 planner 的参数里，同时不让它忘记 SFT 学到的个性化和教学法。"

0. 为什么要做 GRPO？（动机）

SFT 后的 planner 已经能模仿 MAP-PPL 里的金标 plan，但自回归生成本身不保证执行正确——长尾 query 上仍然会冒出循环依赖、非法工具调用、schema 错误。

PersonalPlan 1.0 用的是推理时反思循环（生成→reflector 检查→重写），三个缺点：

  1. 部署贵（每次推理多跑好几遍）
  2. reflector 自己有偏差就把错误带进去了
  3. 近期 reasoning-RL 的研究表明，把检验信号训进参数里比每次推理时再做检验更好

所以 2.0 的思路：把 reflector 干的活搬到训练阶段，用 RL 让 planner 一次就生成对。

1. 奖励设计：三个互补组件 + 一道硬门

(a) Verifiable Structural Reward $R^{\text{struct}}$（对应 challenge iii：结构正确性）

完全规则可验证，不让任何神经裁判插手——因为结构信号最容易被 LLM judge 骗。

$$R^{\text{struct}}_i = w_1\,\mathbb{I}[\text{DAG}(G_i)] + w_2\,\text{DC}_i + w_3\,\text{ATR}_i + w_4\,\text{GED-sim}(G_i, G_i^{\star})$$

四个统计量：

  • 是否是 DAG（拓扑能不能排序）
  • DC：依赖完整性（步骤间该有的依赖有没有写全）
  • ATR：agent–工具相关性（分给某 agent 的工具是不是它该用的）
  • GED-sim：和金标 plan 的图编辑相似度
设计原则：规则不会被花言巧语骗，结构这种最容易作弊的信号必须钉死在规则上。

(b) Counterfactual Personalization Reward $R^{\text{pers}}$（对应 challenge i：个性化）

朴素 LLM judge 一打"够不够个性化"，模型立刻学会塞 persona 关键词作弊。论文换了一个非常巧妙的设计：

同一个 query，分别用真 profile $p$ 和随机替换的假 profile $\tilde p$ 各生成一个 plan：

$$R^{\text{pers}}_i = \underbrace{\text{GED-sim}(\mathcal{P}_i^{(p)}, \mathcal{P}^{\star}_{(p)})}_{\text{真 profile 的 plan 像不像金标}} - \underbrace{\text{GED-sim}(\mathcal{P}_i^{(p)}, \mathcal{P}_i^{(\tilde p)})}_{\text{换 profile 后两个 plan 像不像}}$$

第二项是关键：如果模型只在 plan 里塞了几句"考虑到你是新手……"但结构没变，那换 profile 后两个 plan 还是几乎一样，第二项接近 1，奖励就上不去。只有真的根据 profile 改变了 agent 组合和步骤粒度，奖励才高。

设计原则：用"反事实对照"代替"绝对判断"。问的不是"它好不好"，而是"换个学习者它会不一样吗"。

(c) Prerequisite-Aware Pedagogy Reward $R^{\text{ped}}$（对应 challenge ii：教学法）

教学法既有客观部分（先修关系）也有主观部分（讲解粒度），所以拆成硬软两半：

硬部分（可验证）：从 MAP-PPL 预先建一个概念先修图 $K$（比如"变量作用域 $\preceq_K$ 闭包"），计算 plan 里所有步骤对的先修兼容率：

$$R^{\text{ped,hard}}_i = \frac{|\{(s_a, s_b) \in \mathcal{O}_i: c(s_a)\preceq_K c(s_b)\}|}{|\mathcal{O}_i|}$$

软部分（不能用规则的部分用 LLM judge，但加了护栏）：

  • 顺序随机化（防 position bias）
  • judge ensembling 取中位数（防单个 judge 偏差）

$$R^{\text{ped,soft}}_i = \operatorname{med}_{j\le J}\{\operatorname{Judge}_j(\mathcal{P}_i^{(p)} \succ \mathcal{P}^{\star}_{(p)})\}$$

合并时硬的权重大、软的权重小（$\lambda \in [0.7, 0.9]$）：

$$R^{\text{ped}}_i = \lambda\,R^{\text{ped,hard}}_i + (1-\lambda)\,R^{\text{ped,soft}}_i$$

设计原则：能用规则就用规则，必须用神经裁判时就让它当配角。

(d) Hard Gate $R^{\text{hard}}$（硬门）

对于不可挽回的违规（schema 错、有环、工具不存在），不是扣一点分，而是一票否决：

$$R^{\text{hard}}_i = -\eta\,\mathbb{I}[\text{SchemaInvalid} \lor \text{HasCycle} \lor \text{InvalidToolCall}]$$

设计原则：违章不能用其他优点凑回来。防止模型靠"个性化加分"换"结构 bug"。

2. 奖励合并：组内 z-score 归一化

四个奖励量纲完全不一样（GED-sim 在 [0,1]，硬门是 $-\eta$）。论文先在同一个 prompt 的 $G$ 条采样里做 z-score 归一化，再线性加权：

$$R_i = w_s\,\bar z(R^{\text{struct}}_i) + w_p\,\bar z(R^{\text{pers}}_i) + w_e\,\bar z(R^{\text{ped}}_i) + R^{\text{hard}}_i$$

默认权重 $w_s:w_p:w_e = 0.55:0.25:0.20$。

设计原则：
  1. z-score 抹平量纲差异，否则方差大的组件会主导梯度
  2. 结构奖励权重最大（0.55），因为它最不易被骗，是主信号
  3. 个性化和教学法是"防回归信号"——SFT 已经把这两个能力装进去了，RL 阶段只是别让它退化
  4. 硬门不归一化，永远保持惩罚的绝对量级

3. GRPO 主目标：策略梯度 + KL 锚

"组相对优势" 替代 critic

每个 prompt 采 $G$ 条轨迹，用组内 mean/std 算 advantage：

$$A_i = \frac{R_i - \mu_R}{\sigma_R + \varepsilon}$$

不用训 value model，便宜、稳定、不会震荡。直觉：不需要知道绝对好坏，只需要知道这次比同一道题的其他采样好还是差。

Clipped surrogate + KL 锚到 SFT（论文的关键改造）

$$\mathcal{L}_{\text{GRPO}}(\theta) = -\mathbb{E}\Bigl[\tfrac{1}{G}\sum_i \tfrac{1}{|o_i|}\sum_t \min\bigl(r_{i,t}A_{i,t},\,\operatorname{clip}(r_{i,t},1-\epsilon,1+\epsilon)A_{i,t}\bigr)\Bigr] + \beta\,\mathbb{D}_{\text{KL}}[\pi_\theta\,\|\,\pi_{\text{SFT}}]$$

注意 KL 项钉死在 $\pi_{\text{SFT}}$，而不是常规 GRPO 用的"上一次旧策略"$\pi_{\theta_{\text{old}}}$。

设计原则：SFT 已经把个性化和教学法装进模型了，RL 只是来补"结构正确"的短板。所以要给 RL 戴个"安全绳"——可以往前走，但不许跑远到忘了之前学的东西。这就是为什么 KL 锚点必须是固定的 SFT 模型，不是移动的旧策略。

4. Segment-wise Credit Assignment（分段信用分配）

一个 plan 由 agent 段、subtask 段、step/dependency 段拼接而成，每段对应不同的奖励。如果对所有 token 用同一个 $A_i$，方差大的结构奖励会淹没 agent token 的梯度。

论文的做法是按 token 所属段路由 advantage：

$$A_{i,t} = \begin{cases} \bar z(R^{\text{pers}}_i) & t \in \mathcal{A}_i \text{ (agent 段)} \\ \bar z(R^{\text{ped}}_i) & t \in \mathcal{T}_i \text{ (subtask 段)} \\ \bar z(R^{\text{struct}}_i) & t \in \mathcal{S}_i \cup \mathcal{O}_i \text{ (step/dep 段)} \end{cases}$$

硬门项 $R^{\text{hard}}_i$ 均匀加到所有段（违规是整条轨迹的问题）。

设计原则：谁的活算谁的账。让每个段的 token 只接收和它直接相关的奖励信号，避免互相干扰。

5. 五道反作弊护栏

护栏防什么
(a) Judge 随机化 + 训测异家族训练用一个 family 的 judge，测试用另一个 family。防止策略只迎合某一类 judge 的偏好
(b) 10% 对抗轨迹注入故意塞超长、关键词堆砌、假引用的 plan，要求奖励必须 ≤ 0。直接测试奖励是不是真的不能被骗
(c) Proxy-vs-Gold 监控曲线跟踪"训练奖励"和"人工抽检/结构指标"两条曲线，一旦训练奖励还在涨但人工分数停了，就早停——这是 reward hacking 的典型信号
(d) Loss-mask 到目标 token梯度只更新 action / 结构目标 token，prompt 和 observation 不参与
(e) 自适应 $\beta$一旦策略离 SFT 飘超过预算，自动把 KL 系数调大拽回来

整个 GRPO 设计的三条主线

  1. 奖励维度上的抗作弊：能用规则就用规则（verifiable），不能用规则的就用反事实对照（counterfactual），用神经裁判时就让它当配角并加 ensembling
  2. 优化算法上的稳健：用组相对优势省掉 critic，KL 锚到 SFT 而不是 $\pi_{\text{old}}$，防止 RL 忘掉 SFT 学到的能力
  3. 梯度流动上的精细：奖励按段路由 advantage，让每种 token 只接收它该接收的信号；硬违规一票否决，不让局部优点把整体 bug 救回来
USER 为什么 a 结构正确性还需要 GRPO 而不是在 SFT 阶段就能很好解决

本质问题："为什么 token-level 模仿学不会全局结构？"

1. SFT 优化的是 token-level，结构正确性是 trajectory-level（最核心的原因）

SFT 的损失是：

$$\mathcal{L}_{\text{SFT}} = -\sum_t \log P(y_t \mid x, y_{

这是个局部、逐 token的目标——它只问"下一个 token 是不是和金标一样"。

但 "是不是 DAG"、"工具调不调用得起来"、"schema 合不合法"，这些是整个输出生成完之后才能判断的全局属性。

举个具体例子：

  • 一个 15 步的 plan，14 步都对，只有第 12 步的 depends_on 字段引用了一个不存在的 step ID → SFT loss 几乎完美，但这条 plan 不可执行
  • 一个 plan 的 JSON 完全合法、agent 名字也对，但依赖关系刚好形成一个环 → token-level 看不出问题，但拓扑排序直接挂掉

SFT 永远不会"惩罚成环"这件事，因为它根本不知道有没有环——它只比对 token。

对比 GRPO 用的奖励：

$$R^{\text{struct}} = \mathbb{I}[\text{DAG}] + \text{DC} + \text{ATR} + \text{GED-sim}$$

这是先把整条 plan 生成完，跑一遍图算法算出来的——它直接测量我们想要的属性。

一句话：SFT 没法把"DAG"这种组合约束写进损失函数；GRPO 可以。

2. Exposure Bias（暴露偏差）：训练和推理的输入分布不一样

SFT 训练时，模型生成第 $t$ 个 token 时条件在金标的前 $t-1$ 个 token 上（teacher forcing）。

但推理时，模型条件在自己生成的前 $t-1$ 个 token 上——一旦前面出错，错误会滚雪球式累积。

这在 plan 生成上尤其致命：

  • 第 3 步如果稍微写偏了，第 12 步引用第 3 步时就可能引用错
  • 然后第 15 步基于第 12 步的（错误）输出再编依赖，整条 chain 就崩了

理论上的复杂度差异：

  • SFT 推理误差 $\sim O(T^2)$（$T$ 是序列长度）
  • 有 on-policy 修正的 RL 误差 $\sim O(T)$

GRPO 是 on-policy 的——它采样自己生成的轨迹然后打分，所以模型在训练时就见过自己的错误并学会从中恢复。SFT 完全见不到。

一句话：SFT 只见过"标准答案下一步"，没见过"自己出错之后该怎么办"。

3. SFT 没有"负样本"，只能模仿不能避错

SFT 只见过金标 plan（都是正确的）。它从来没见过：

  • "这个 plan 有环 → 坏"
  • "这个 plan 工具用错了 → 坏"
  • "这个 plan schema 违法 → 坏"

它只能正向模仿，没法反向回避。

GRPO 的逻辑完全相反：

  • 每个 prompt 采 $G$ 条轨迹，有好有坏
  • 好的（$R$ 高）给正梯度，坏的（$R$ 低）给负梯度
  • 硬门项 $R^{\text{hard}} = -\eta$ 还专门给违规的 plan 直接打负分
一句话：SFT 是"抄优等生作业"，GRPO 是"做完一堆题然后被改卷"——后者才能学到错题模式。

4. MAP-PPL 数据稀疏，长尾 query 上 SFT 没东西可模仿

MAP-PPL 只有 3,608 条 plan。对于训练分布里常见的 query，SFT 后模型生成的 plan 结构基本没问题。

但对于长尾 query（很冷门的组合，比如"在 Rust 里调用 ONNX 模型并部署到 WASM"），训练集里几乎没有相似的 plan。SFT 只能靠泛化——而结构正确性恰好是泛化最差的部分，因为它依赖精确的 ID 匹配和拓扑约束。

GRPO 不需要"长得像金标"。它只需要"生成出来的 plan 跑得起来"——而这个判定是规则化的，不依赖训练数据覆盖。

一句话：金标只有 3,608 条，但 DAG 检查器可以对任何 plan 都给一个明确的对错信号。GRPO 把这个信号利用上了，SFT 用不上。

5. SFT loss 和 structural reward 在数学上根本不对齐

考虑两个 plan A 和 B：

Plan APlan B
和金标的 token 重合度99%70%
SFT loss很低（好）很高（差）
是不是 DAG❌ 有环✅ 合法
工具调用✅ 都对✅ 都对
真实可用性0（跑不起来）1（能跑）

SFT 会把 A 评得很高、B 评得很差。但我们要的是 B。

也就是说：SFT 损失最小化的解，并不是结构正确率最高的解——两者是不同的目标函数。

一句话：SFT 在优化的是代理目标（imitation），GRPO 在优化的是真实目标（executability）。

核心矛盾你已经点出来了

GRPO 本身在一个 prompt 下采 $G$ 条轨迹；但 $R^{\text{pers}}$ 需要另一组用假 profile $\tilde p$ 生成的 plan 来当对照——这两层采样怎么协调？

答案的关键是一个身份区分：

真 profile 的 plan 是"训练轨迹"（要算梯度），假 profile 的 plan 是"参考 rollout"（只用来打分，不算梯度）。

1. 把每个 prompt 的"采样束"重新定义

普通 GRPO 一个 prompt 采 $G$ 条轨迹；本论文一个 prompt 实际上要采 $G + k$ 条：

角色数量是否进入策略梯度用途
真 profile 主样 $\mathcal{P}_i^{(p)} \sim \pi_\theta(\cdot \mid q, p)$$G$ 条✅ 进 lossGRPO 的训练对象
假 profile 对照样 $\mathcal{P}^{(\tilde p)} \sim \pi_\theta(\cdot \mid q, \tilde p)$$k$ 条（通常 $k=1$）❌ no_grad / detach只作为 $R^{\text{pers}}$ 计算时的参照

所以一个 prompt 的 group 只有那 $G$ 条真 profile 主样——组内 mean/std、advantage 归一化、KL 锚都只在这 $G$ 条上算。

2. 具体的训练循环长这样（伪代码）

for batch in dataloader:
    for (q, p, P_gold) in batch:
        # ───── Step 1: 真 profile 主采样 (要梯度) ─────
        with torch.enable_grad():
            P_true = [π_θ.sample(q, p) for _ in range(G)]   # G 条

        # ───── Step 2: 假 profile 对照采样 (不要梯度) ───
        p_tilde = sample_other_profile(batch, exclude=p)
        with torch.no_grad():
            P_cf = π_θ.sample(q, p_tilde)   # k=1 条就够

        # ───── Step 3: 算每条主样的奖励 ──────────────
        R = []
        for P_i in P_true:
            r_struct = struct_reward(P_i)                              # 规则
            r_ped    = ped_reward(P_i)                                 # 规则+judge
            r_pers   = ged_sim(P_i, P_gold) - ged_sim(P_i, P_cf)       # 反事实
            r_hard   = hard_gate(P_i)
            R.append(combine(r_struct, r_pers, r_ped, r_hard))

        # ───── Step 4: 组内归一化 + 标准 GRPO 更新 ─────
        A = (R - mean(R)) / (std(R) + eps)
        loss = grpo_clipped_surrogate(P_true, A) \
             + β * KL(π_θ(·|q,p) ‖ π_SFT(·|q,p))

关键点：

  • $G$ 条主样共享同一个 $\mathcal{P}^{(\tilde p)}$——所有主样都和这同一条"假 profile 对照"算 GED-sim。每个 query 只多采 1 条额外 rollout，相对 $G$ 条主样几乎可以忽略（$G=8$ 时多 12% 的采样开销）。
  • 假 profile 的 rollout 在 torch.no_grad() 下生成，它的 logits 不进入计算图。

3. 假 profile $\tilde p$ 怎么来？

方法做法优缺点
(a) Batch 内换在同一个 batch 里随机抽另一个样本的 profile 当 $\tilde p$几乎零成本；但有时 $\tilde p$ 和 $p$ 长得太像
(b) Profile bank预先建一个"对极学习者"profile 库，每个 $p$ 配一个最远的 $\tilde p$控制力强；要预先维护一个库
(c) 类别翻转把 profile 关键字段（skill_level, goal）反向，然后重生成其他字段反事实最干净；实现稍复杂

最常用的是 (a)，这是自然产生 in-batch negative，和对比学习里的做法完全一致。

4. 这条反事实样本的 reward "去哪了"？

假 profile 的 plan 本身不参与任何 reward 计算和梯度更新。它的唯一作用是作为 $R^{\text{pers}}$ 公式里的 $\mathcal{P}_i^{(\tilde p)}$，被真 profile 主样减一下：

$$R^{\text{pers}}_i = \underbrace{\text{GED-sim}(\mathcal{P}_i^{(p)},\, \mathcal{P}^\star_{(p)})}_{\text{真样像不像金标}} \;-\; \underbrace{\text{GED-sim}(\mathcal{P}_i^{(p)},\, \mathcal{P}^{(\tilde p)})}_{\text{真样和反事实样像不像}}$$

  • 真样产生 reward + 梯度 ✅
  • 反事实样只产生一个数值（被 GED-sim 拿来比较），没有 reward，没有梯度 ❌

5. 这一切怎么和 segment-wise credit 接上

$R^{\text{pers}}$ 经过组内 z-score 后，只贡献给真 profile 主样里"agent 段" token 的梯度。

  • 反事实采样只影响 agent 段的 advantage
  • 结构段和 subtask 段的梯度完全不受反事实采样的"$k$ 条额外 rollout"干扰
  • 反事实采样 ≈ 给 agent 段的 advantage 做一次"对照实验校正"

6. 一些容易忽略的工程坑

(a) KL 锚怎么算？

$\mathbb{D}_{\text{KL}}[\pi_\theta \| \pi_{\text{SFT}}]$ 应该只在真 profile 主样上算——不要在 $\mathcal{P}^{(\tilde p)}$ 上算 KL。

(b) 防止"diverge 作弊"

模型会不会学到"让所有 profile 生成完全不同的 plan"来人为放大第二项？不会，因为第一项 $\text{GED-sim}(\mathcal{P}_i^{(p)}, \mathcal{P}^\star_{(p)})$ 钉死它必须靠近金标。模型只有改对方向（按 profile 真的调整结构）才能同时拉高第一项、压低第二项。

(c) 采样效率优化

backbone 跑 $(q, p)$ 和 $(q, \tilde p)$ 时，$q$ 那段的 KV cache 可以复用——只需要重算 profile 部分的 prefix。

(d) 是否每条主样都配一条独立 $\tilde p$？

$k=1$ 通常就够了——所有 $G$ 条主样共享一个 $\mathcal{P}^{(\tilde p)}$。如果想降噪，可以采 $k=2{\sim}4$ 条 $\tilde p$，把第二项改成 $\frac{1}{k}\sum$ 的平均 GED-sim。

一句话总结

"训练的 group 还是只有 $G$ 条真 profile 主样"——反事实 rollout 是放在 no_grad 块里的 reference，它的唯一身份是 $R^{\text{pers}}$ 公式里那个被减掉的 baseline，既不出现在 advantage 归一化里，也不出现在 KL 锚里。

多出来的 $\sim 12\%$ 推理开销，换来一个不能被"塞 persona 关键词"作弊的个性化奖励。

USER Prerequisite-Aware Pedagogy Reward 这个你也解释一下

Prerequisite-Aware Pedagogy Reward 完整解读

这个奖励要回答的问题是："这个 plan 教得对吗？" 但"教得对"既有客观的一面（先教概念 A 再教 B）也有主观的一面（讲解的密度、铺垫得是否合适）。

论文的核心设计哲学是 "硬软分离，硬为主、软为辅"：

$$R^{\text{ped}}_i = \lambda \cdot R^{\text{ped,hard}}_i + (1-\lambda) \cdot R^{\text{ped,soft}}_i,\quad \lambda \in [0.7, 0.9]$$

1. Hard 部分：概念先修图 $K$ + 兼容率（最关键的设计）

(a) 先修图 $K = (V_K, E_K)$ 是什么？

这是一个离线预先建好的有向图：

  • 节点 $V_K$：编程教学里的概念，比如 "变量作用域"、"闭包"、"函数定义"、"REST 原则"、"HTTP 状态码"……
  • 边 $E_K$：先修关系。$a \preceq_K b$ 表示"学 $b$ 之前应该先学 $a$"

比如：

变量定义  →  变量作用域  →  闭包  →  高阶函数
环境配置  →  包管理     →  虚拟环境  →  跑代码
HTTP 基础 →  REST 原则  →  路由设计  →  鉴权

(b) 这张图怎么建出来？

论文里写的是"From the MAP-PPL ground-truth plans we precompute"——$K$ 不是手工写的，而是从训练集里自动归纳的：

方法做法
频次统计扫所有金标 plan，如果"概念 $a$"在"概念 $b$"前出现 ≥ 阈值次（且很少反过来），就建一条 $a \to b$ 的边
聚合 + 闭包先建一阶共现图，然后做传递闭包 / DAG 剪枝去掉冲突边

关键：$K$ 是全局共享、一次性建好的。GRPO 训练循环里只查它，不更新它。

(c) 步骤 → 概念的映射 $c(\cdot)$

$c(s)$ 把一个具体步骤映射到 $K$ 的某个概念节点。工程上：

  • 关键词/实体匹配：用预先建好的 concept → keyword 词典
  • Embedding 最近邻：把 $V_K$ 节点描述向量化，取 cosine 最近
  • 小模型分类器：训一个 lightweight classifier

(d) 兼容率的具体含义

$$R^{\text{ped,hard}}_i = \frac{|\{(s_a, s_b) \in \mathcal{O}_i : c(s_a) \preceq_K c(s_b)\}|}{|\mathcal{O}_i|}$$

  • 分母 $|\mathcal{O}_i|$：plan 里所有依赖边的总数
  • 分子：这些边里"先修方向正确"的边数

例子：plan 里写了"步骤 5 (闭包实现) 依赖 步骤 3 (变量作用域)"——查 $K$ 发现"变量作用域 $\preceq_K$ 闭包"，✅ 合法。

设计原则：教学顺序错不错是个客观事实——这部分完全规则可验证。

2. Soft 部分：成对 LLM judge（保底，权重小）

(a) 为什么还要 soft 部分？

有些"教学法好不好"的维度没法用规则量化：

  • "讲解粒度"够不够细
  • 是否在合适的地方插入了"复习/巩固"的步骤
  • 学习者能不能跟上节奏

(b) 公式

$$R^{\text{ped,soft}}_i = \operatorname{med}_{j \le J}\{\operatorname{Judge}_j(\mathcal{P}_i^{(p)} \succ \mathcal{P}^\star_{(p)})\}$$

  • $\mathcal{P}_i^{(p)} \succ \mathcal{P}^\star_{(p)}$：成对比较——问 judge "学生 plan 比金标 plan 更教学法好吗？"
  • $\operatorname{Judge}_j$：第 $j$ 个 judge 给出的偏好分
  • $j \le J$：有 $J$ 个不同的 judge（ensembling）
  • $\operatorname{med}$：取中位数，不是平均

(c) 关键护栏

  1. 成对比较 ≠ 绝对打分：对 calibration 不敏感
  2. 顺序随机化：消除 position bias
  3. Judge ensembling：$J$ 个 judge，取中位数
  4. 训测异家族：训练 vs 评估使用不同模型 family

(d) 为什么权重小（$1-\lambda \in [0.1, 0.3]$）

"Keeping the hackable judge subordinate to the verifiable component."

LLM judge 是整个奖励系统里最容易被作弊的组件，所以即使加了一堆护栏，论文还是把它的权重压到 ≤ 30%。

3. 与硬/软对应的反作弊逻辑

Reward主要作弊面论文怎么防
$R^{\text{struct}}$LLM judge 容易被骗全用规则，零 judge
$R^{\text{pers}}$persona 关键词作弊反事实对照，差值结构
$R^{\text{ped}}$judge 主观、有偏拆硬软，硬为主（70-90%），软用 ensembling+顺序随机化

4. 在 GRPO 里怎么实现

# 离线预处理（训练前做一次）
K = build_prerequisite_graph_from(MAPLE_gold_plans)
c = load_concept_mapper()
judges = [load_judge(family_A_model_1, prompt_v1),
          load_judge(family_A_model_2, prompt_v2),
          load_judge(family_B_model_1, prompt_v3)]

# 在 GRPO 训练循环里
for batch in dataloader:
    for (q, p, P_gold) in batch:
        P_true = [π_θ.sample(q, p) for _ in range(G)]
        for P_i in P_true:
            # ---- Hard 部分：直接查图 ----
            edges_in_O = P_i.dependency_edges()
            ok = sum(1 for (s_a, s_b) in edges_in_O
                       if (c(s_b), c(s_a)) in K.edges())
            r_ped_hard = ok / max(len(edges_in_O), 1)

            # ---- Soft 部分：成对 judge + 顺序随机化 + 中位数 ----
            scores = []
            for judge_j in judges:
                if random.random() < 0.5:
                    s = judge_j(P_i, P_gold)
                else:
                    s = 1 - judge_j(P_gold, P_i)
                scores.append(s)
            r_ped_soft = median(scores)

            r_ped = λ * r_ped_hard + (1 - λ) * r_ped_soft   # λ ≈ 0.8

5. 怎么和 segment-wise credit 接上

$R^{\text{ped}}$ 被路由到 subtask 段——也就是说先修兼容率算的是 step 之间的依赖（$\mathcal{O}_i$），但奖励只回流给 subtask token 的梯度。

为什么？

  • subtask 段决定了整个 plan 的"教学骨架"
  • step 级别的依赖正确性已经被 $R^{\text{struct}}$ 抓住了
  • 把"先修兼容率"路由到 subtask 段 = "你拆 subtask 的时候要按照知识的逻辑顺序拆"
奖励的计算是基于 step 间依赖 $\mathcal{O}_i$，但奖励的回传是给 subtask token——这是 segment-wise credit 的关键巧思。

一句话总结

教学法奖励 = "概念顺序对不对（规则、80%权重）" + "讲解风格好不好（ensemble judge、20%权重）"。

硬部分把可形式化的教学法规则离线编码成一张先修图 $K$，训练时 0 神经裁判、几乎 0 成本；软部分用成对比较+顺序随机化+多 judge 中位数把 LLM judge 的偏差压到最低，且权重压到 30% 以内。

最后这个奖励被路由到 subtask 段的 token，专门优化"教学骨架"的搭建。

算分时怎么把 verdict 变成标量

def soft_score(subscores, verdict, w_sub=0.6, w_verdict=0.4):
    sub = mean([{"A":1, "Tie":0.5, "B":0}[s] for s in subscores.values()])
    ver = {"A":1, "Tie":0.5, "B":0}[verdict]
    return w_sub * sub + w_verdict * ver

多 Reward 联合优化是个两层问题

很多人误以为多 reward 优化就是简单的加权求和，但在论文这套设计里"权重分配"和"怎么同时优化"在两个不同的层级上发挥作用：

层级任务谁来做权重在哪
Level 1：Reward 融合把 4 个标量奖励合成 1 个 trajectory-level 信号Hard Gate + Group-wise Normalization 段$w_s, w_p, w_e$ 出现在加权和里
Level 2：梯度路由把信号回传到具体 token 上Segment-wise Credit Assignment 段隐式地由"段"决定，不再显式加权

Level 1：Reward 融合——权重出现在这里

四个奖励先各自归一化、再线性加权：

$$R_i = w_s\,\bar z(R^{\text{struct}}_i) + w_p\,\bar z(R^{\text{pers}}_i) + w_e\,\bar z(R^{\text{ped}}_i) + R^{\text{hard}}_i$$

(a) 先 z-score、再加权——保证权重真的等于"优先级"

如果不归一化直接加权，会出问题：

  • $R^{\text{struct}}$ 范围在 $[0, 4]$
  • $R^{\text{pers}}$ 范围在 $[-1, 1]$
  • $R^{\text{ped}}$ 范围在 $[0, 1]$

就算 $w_s = 0.1$，结构奖励的绝对量级仍然能淹没其他两个——权重就失去了"我想给谁多少话语权"的语义。

z-score 把每个组件强制压到同一个均值 0、方差 1 的尺度——这时 $w_s = 0.55$ 真的等于"55% 的话语权"。

没有这一步，权重只是装饰品。

(b) 默认权重 $w_s : w_p : w_e = 0.55 : 0.25 : 0.20$ 的逻辑

Reward抗作弊等级SFT 学得动吗应得权重
$R^{\text{struct}}$最高（纯规则）学不动（全局约束）最大 0.55
$R^{\text{pers}}$高（反事实差值）SFT 已经会大半中 0.25
$R^{\text{ped}}$中（80% 规则 + 20% judge）SFT 已经会大半小 0.20

口诀：

最不易作弊 + SFT 学不到的，给最大权重。

(c) Hard Gate 故意不归一化——它是约束不是偏好

$R^{\text{hard}}_i = -\eta\,\mathbb{I}[\text{违规}]$，没有 $w_h$，没有 $\bar z$。

为什么？如果 z-score 化它：

  • 整个 batch 都没违规 → 归一化后是 0，根本没惩罚
  • 一个 batch 全违规 → 归一化后也变 0，"大家一起违规就不算违规了"

所以硬门必须保持绝对量级。

z-score 化的是"偏好信号"；保持绝对量级的是"硬约束"。

Level 2：梯度路由——权重在这里"消失"

按 GRPO 流程，融合好的 $R_i$ 应该走标准 advantage，所有 token 共享同一个 $A_i$。但论文不这样做——它做了 segment-wise credit assignment：

$$A_{i,t} = \begin{cases} \bar z(R^{\text{pers}}_i) & t \in \mathcal{A}_i \\ \bar z(R^{\text{ped}}_i) & t \in \mathcal{T}_i \\ \bar z(R^{\text{struct}}_i) & t \in \mathcal{S}_i \cup \mathcal{O}_i \end{cases} \;+\; R^{\text{hard}}_i$$

注意公式里没有 $w_s, w_p, w_e$！

这是因为：每个 token 实际上只接收一个 reward 的信号——agent token 只看 $R^{\text{pers}}$，step token 只看 $R^{\text{struct}}$。它们之间根本不在同一个 token 上竞争。

为什么"权重消失"反而是优雅的？

考虑反例：如果用共享 $A_i$ 的标准 GRPO，所有 token 的梯度方向都被加权和决定：

  • 一条 plan 结构对、个性化差 → 整体 $A_i$ 大 → agent token 也被推向"保持当前个性化"的方向（错的奖励信号）
  • 一条 plan 结构错、个性化对 → 整体 $A_i$ 小 → agent token 也被打压（明明 agent 段做得对）

Segment-wise routing 解决了这个：

轨迹特征agent token 的梯度方向step token 的梯度方向
结构✅、个性化❌下降（个性化差）上升（结构好）
结构❌、个性化✅上升（个性化好）下降（结构差）

每个 token 只对自己负责的那部分 reward 做出响应。

那 Level 1 的权重还有什么用？

(1) 用作轨迹排序和早停判据

  • anti-hacking safeguard 监控 proxy-vs-gold 曲线——proxy 就是这个加权 $R_i$
  • 同一个 prompt 内 $G$ 条轨迹排序、挑出最佳/最差用于人工抽检

(2) 用作 Pareto curve 的旋钮

"By varying the relative weights of the three reward components, the same trained planner exposes a controllable Pareto curve..."

具体来说：

  • 训练前选一组 $(w_s, w_p, w_e)$ → 得到一个偏重某维度的 planner
  • 多组权重 → 多个 checkpoint → 画出 Pareto 前沿

(3) 用作 segment-wise 实现中的隐式重要性采样修正

有的实现里会把 $w_*$ 拉回到 segment-wise advantage 上：

$$A_{i,t} = \begin{cases} \alpha_p \cdot \bar z(R^{\text{pers}}_i) & t \in \mathcal{A}_i \\ \alpha_e \cdot \bar z(R^{\text{ped}}_i) & t \in \mathcal{T}_i \\ \alpha_s \cdot \bar z(R^{\text{struct}}_i) & t \in \mathcal{S}_i \cup \mathcal{O}_i \end{cases}$$

其中 $\alpha_*$ 用来补偿不同 segment 的 token 数差异——例如 step 段 token 通常是 agent 段的 5-10 倍。如果想精确控制各段贡献，可以让 $\alpha_* \propto w_* / |\text{segment}|$。

把整个联合优化串起来（伪代码）

for batch in loader:
    for (q, p, P_gold) in batch:
        # ============= 采样 =============
        traj = [π_θ.sample(q, p) for _ in range(G)]
        with no_grad: traj_cf = π_θ.sample(q, p_tilde)

        # ============= 4 个 reward 各自打分 =============
        for i in range(G):
            r_struct[i] = struct_reward(traj[i])
            r_pers[i]   = ged_sim(traj[i], P_gold) - ged_sim(traj[i], traj_cf)
            r_ped[i]    = λ*ped_hard(traj[i]) + (1-λ)*ped_soft(traj[i])
            r_hard[i]   = hard_gate(traj[i])

        # ============= Level 1: 融合（用于排序/监控） =============
        R = w_s*z_score(r_struct) + w_p*z_score(r_pers) \
          + w_e*z_score(r_ped) + r_hard
        # ↑ 这个 R 用于 proxy-vs-gold 监控和早停判据，不直接进 loss

        # ============= Level 2: segment-wise routing（进 loss） =============
        for i in range(G):
            for t in range(len(traj[i])):
                seg = which_segment(traj[i], t)
                if   seg == 'agent':    A[i, t] = z_score(r_pers)[i]
                elif seg == 'subtask':  A[i, t] = z_score(r_ped)[i]
                else:                   A[i, t] = z_score(r_struct)[i]
                A[i, t] += r_hard[i]   # 硬门均匀加到每个 token

        # ============= GRPO 主目标 =============
        loss = -clipped_surrogate(traj, A, π_θ, π_θ_old) \
             + β * KL(π_θ ‖ π_SFT)
        loss.backward(); optimizer.step()

        # ============= Anti-hacking: 监控 R vs gold =============
        if proxy_rising_but_gold_plateau(R, gold_metrics):
            early_stop()
        if KL(π_θ ‖ π_SFT) > budget:
            β *= 1.5   # 自适应调大

一句话总结

多 reward 不是"合成一个标量然后优化"——是"按段路由各 reward 各管一段 token，组合起来再加一个公共的硬门"。

权重 $w_s, w_p, w_e$ 只在融合层（Level 1）出现，用途是：

  1. 给 proxy 信号做线性投影，用于监控/排序；
  2. 当作 Pareto 旋钮，训前一调就能换一组 trade-off；
  3. 可选地反馈回 segment advantage，做 token 数补偿。

真正进梯度的（Level 2）是 z-score 后的每段 reward + 硬门。两类信号天然不冲突——它们根本不在同一个 token 上抢位置。

这种设计的本质是 "用结构性 routing 替代向量加权"——加权适合标量目标的多目标优化（比如 Pareto 搜索），routing 适合结构化输出（plan 由不同段组成）的多目标优化。论文在这两件事上做了对应的工具选择，避免了传统多目标 RL 里"权重一改，所有 token 都跟着摇摆"的不稳定性。
