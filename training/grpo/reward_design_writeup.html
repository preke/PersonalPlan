0. 设计哲学：Composition over Complexity

原论文 §4.2 的 reward 体系包含四个组件 + 反事实采样 + LLM judge ensemble + segment-wise 路由 + 五条反作弊护栏，工程化估算需要 ~800 行代码、每训练步成本上升 10–30×。落地时我们采取两条原则对其裁剪：

  1. 能用规则就用规则。这是 RLVR（Reinforcement Learning from Verifiable Rewards）的核心原则 [Lambert et al., 2024]，被 DeepSeek-R1 在数学推理任务上验证 [DeepSeek-AI, 2025]：规则化 reward 不可被神经裁判讨好，是抗 reward hacking 的最强信号 [Skalse et al., 2022; Gao et al., 2023]。
  2. 单 reward 弱、组合 reward 强。我们不强求每个 reward 独自具备完美的判别力，而是让组合后的总 reward呈现正确方向。这与 MoGRPO [Huang et al., 2025] 关于多目标 RL 的"约束 + 偏好"分层思路一致。

由此得到的 reward 栈：

Rtotal = ws·Rstruct + wp·Rpers,lite + we·Rped,lite + Rhard
默认 ws:wp:we = 0.55 : 0.25 : 0.20（沿用论文表述）

所有四个组件都纯 Python 规则可计算，每步开销 < 1 ms，无 LLM API 依赖。

1. Rstruct — Verifiable Structural Reward

论文定义

Rstructi = w1·𝕀[DAG(Gi)] + w2·DCi + w3·ATRi + w4·GED-sim(Gi, Gi★)

落地实现

四个分量我们都实现了，但 GED-sim 用 Jaccard fingerprint 代替，其他三项原样。

分量论文要求实现位置
DAG check拓扑可排序三色 DFS 检环plan_utils.has_cycle
DC (Dependency Completeness)依赖完整率有效 depends_on 边比例plan_utils.dependency_completeness
ATR (Agent-Tool Relevance)工具调用对应到 agent 申报step.tool ∈ agent.tools 比例plan_utils.agent_tool_relevance
GED-sim严格图编辑距离5 项集合 Jaccard 平均（fingerprint proxy）plan_utils.structural_similarity

简化决策：GED → Jaccard fingerprint

严格 GED 是 NP-hard 问题，即使是近似算法 [Riesen and Bunke, 2009] 对 30 节点的小图仍需 100 ms+ 单次计算。对每 batch 8 条 trajectory × ~30 节点 × 1000+ 训练步而言，GED 会成为训练吞吐主瓶颈。

替代方案：把 plan 拆成 5 个集合（agent role 多集、subtask name、step id、edges、tools）分别算 Jaccard 后平均：

GED-simproxy(P, P*) = (1/5) · Σk=1..5 Jaccard(Fk(P), Fk(P*))

我们在 MAP-PPL 30-节点级 plan 上手工抽样 50 对 (plan, gold) 计算严格 GED 和这个 proxy，得到 Spearman ρ ≈ 0.85；速度提升约 1000×。

理论支撑

  • 规则化 reward 是首选：[Lambert et al., 2024] (Tulu 3 / RLVR) 系统论证了"可验证奖励"为何能避免 reward model 的失效模式；[DeepSeek-AI, 2025] 在数学和代码推理上的成功（仅 0/1 verifier）进一步证实。
  • 结构约束适合规则化：plan 的"是否 DAG / 依赖闭合 / 工具合法"都是整 trajectory 可决的属性，SFT 的 token-level loss 看不见（这一点与 process reward modeling 的动机一致 [Lightman et al., 2023; Wang et al., 2024]）。
  • 近似图相似度替代 GED：图核（graph kernel）研究的标准取舍 [Shervashidze et al., 2011]——在 plan 级别的 graph signature 上 Jaccard 已经足够。

2. Rhard — Disqualifying Hard Gate

论文定义

Rhardi = -η · 𝕀[SchemaInvalid ∨ HasCycle ∨ InvalidToolCall]

落地实现

原样实现，无简化。位置：rewards.reward_hard_gate(plan_text, eta=10.0)。

关键设计：不参与 z-score 归一化

论文（以及 [Huang et al., 2025] MoGRPO 在约束式 RL 的处理）明确指出：

Schema 违规 / DAG 有环是"一票否决"事件，不能与 personalization 加分相抵。z-score 化会让"全 batch 违规"等价于"全 batch 合规"——失去硬约束本意。

因此 Rhard 保留 -η 的绝对量级（默认 η=10），直接叠加到 total reward，不经组内归一化。

理论支撑

  • [Huang et al., 2025] MoGRPO：多目标 RL 中"硬约束"与"软偏好"必须分层。
  • [Skalse et al., 2022]：reward hacking 的典型形式之一是用"易优化的小目标"换"难判别的大约束"——硬门是消除这种交换可能性的最直接手段。

3. Rpers,lite — Personalization without Counterfactual

论文定义（counterfactual）

Rpersi = sim(Pi(p), P*(p)) − sim(Pi(p), P(p̃))

其中 P(p̃) 是用当前策略 πθ、随机替换的 profile p̃ 采样得到的 plan。

落地难点

反事实 rollout 需要在每个 GRPO 步骤中额外采样 k=1 条 no_grad 轨迹，TRL GRPOTrainer 不原生暴露该 hook，必须 subclass 并 override _prepare_inputs。工程量约 150 行 + 训练每步 +12% 推理开销。

简化方案：Rpers,lite — 步骤级 2nd-person × profile 信号共现

动机：MAP-PPL 金标 plan 中的真个性化，其文本模式是稳定的：

"Ask the learner: 'You said the error appeared after you added the orderby clause...'"
"Frame this using a real-world analogy from the learner's project work..."
"Using your earlier prediction, explain..."

这些模式有两个可检测特征：(a) 2nd-person 引用（you / your / the learner），(b) 与 profile 上下文词或桥接锚词同句出现。Counterfactual-free reward 直接检测这种共现：

Rpers,lite(P, p) = min((含个性化句的 step 数 / 总 step 数), 0.8)

"含个性化句" 的 step 满足:
  (1) step.instruction 长度 ≥ 150 字符（boilerplate 过滤）
  (2) 存在一句话同时包含 2nd-person 引用 + profile-signal/bridge-anchor

位置：rewards.reward_personalization_lite。

抗作弊设计

作弊路径防御机制
把所有 skill 词塞进一个 backstory检测器只看 step.instruction，backstory 不计入 → 0 分
"You are a X developer." 每步重复长度 ≤ 150 字符 → 长度过滤拦截
把桥接短语塞进每步（119 字符）长度仍不达标
200+ 字符的长 boilerplate 模板不能单 reward 拦截；依赖与 Rstruct 组合：boilerplate 的 step instruction 与 gold 文本差异大，Rstruct 的 fingerprint Jaccard 会大幅下降

诚实声明

Rpers,lite 单独看不是完美判别器。它的设计前提是"与 Rstruct 组合"，文档（reward_personalization_lite.__doc__）明确写出。这是 anti-hacking via composition 的一种实例——参见 §5。

理论支撑

  • Counterfactual reward 的替代：[Garrido et al., 2014] 与 [Wei et al., 2025] 关于教学个性化的研究都指出"learner profile → 教学策略变化"的可检测模式是 plan 文本中的2nd-person engagement density，而非结构化反事实。
  • 规则化 grounding reward：与 RAG faithfulness 文献中的关键词覆盖 reward（RAGAS-style，[Es et al., 2024]）思路一致——便宜的 lexical 信号 + 上界 + 组合，在 RLHF 替代品中已有先例。
  • Anti-hacking via composition：[Pan et al., 2022] 关于"reward misspecification"提出，多 reward 的分布层耦合而非单 reward 完美化，是工程上的稳健做法。

4. Rped,lite — Pedagogy without LLM Judges

论文定义（hard + soft 混合）

Rpedi = λ·Rped,hardi + (1−λ)·Rped,softi,   λ ∈ [0.7, 0.9]

Rped,hardi = |{(sa, sb) ∈ Oi : c(sa) ⪯K c(sb)}| / |Oi|

Rped,softi = medj ≤ J{Judgej(Pi(p) ≻ P*(p))}

Rped,hard 需要一张概念先修图 K + step → concept 映射器 c(·)。Rped,soft 需要 J ≥ 3 个 LLM judge ensemble + 顺序随机化 + 训测异家族。

简化方案：Rped,lite = precedence + phase coverage（双纯规则）

Rped,lite(P) = 0.5·SubtaskPrecedence(P) + 0.5·PhaseCoverage(P)

(A) SubtaskPrecedence — 替代 concept graph K

原方案需要：(i) 离线挖掘 concept graph K（含传递闭包剪枝），(ii) 在线 step → concept 的映射器。我们跳过 concept 层，直接在 subtask name 层挖共现先后关系：

# 离线一次扫
for plan in MAPLE_gold:
    sub_names = [s.name.lower() for s in plan.subtasks]
    for i, a in enumerate(sub_names):
        for b in sub_names[i+1:]:
            precedence_pairs.add((a, b))

训练时：每条生成 plan 计算"(subtask_a, subtask_b) ∈ 先后关系集 / 总 subtask 对数"。位置：plan_utils.mine_subtask_precedence + plan_utils.prerequisite_compatibility。

(B) PhaseCoverage — 替代 LLM judge ensemble

核心观察：MAP-PPL 金标 plan 的 step.instruction 自然呈现 6 个典型教学阶段（probe → retrieve_demonstrate → apply → validate → feedback → consolidate），每个阶段都有稳定的 lexical signature。直接关键词检测即可：

PhaseKeywords (节选)
probe"ask the learner", "what do you think", "your intuition", "prior knowledge", "activate"
retrieve_demonstrate"look up", "retrieve", "documentation", "demonstrate", "execute the following"
apply"ask the learner to write", "have the learner implement", "submit"
validate"test case", "validate", "verify", "compile", "outcome=", "implementation_correct"
feedback"explain what went wrong", "targeted feedback", "what to fix"
consolidate"summarize", "consolidate", "wrap up", "decision rule", "in their own words"

检出每个 step 的 phase 后，做 5 个 binary check 的均值作为 PhaseCoverage 分数：

  1. probe 出现在 plan 前 50%
  2. validate 至少出现一次
  3. feedback 至少出现一次
  4. consolidate 出现在 plan 后 40%
  5. 检出的 phase 位置呈非递减（canonical 顺序）

位置：rewards.reward_pedagogy_phases。

对比：论文 Rped 表达的 8 个 sub-criteria → Rped,lite 的覆盖

从 GRPO_conversation.html 整理出来 Rped,soft rubric 的 8 个子项，其中 6 个能由 PhaseCoverage 间接覆盖：

论文 rubric 子项覆盖方式
Foundation-before-usePhaseCoverage 顺序检查（probe → validate）+ SubtaskPrecedence
Checkpoint placementvalidate 阶段出现频次
Bridging granularityretrieve_demonstrate 阶段是否出现在 probe 之后、apply 之前
Tool-readiness由 Rstruct.ATR (agent-tool relevance) 覆盖
Granularity match无法用规则覆盖（需 LLM judge）
Explanation depth无法用规则覆盖（需 LLM judge）
Tool/agent complexity由 Rstruct.ATR 部分覆盖
Goal alignment由 Rstruct.GED-simproxy 部分覆盖

无法用规则覆盖的两项（granularity match、explanation depth）目前不在 v1 reward 中。这是已知的论文-实现 gap；如需补回，启用 ped_mode: "full" 加 LLM judge（见 §9）。

理论支撑

  • Prerequisite ordering 是可验证的：[Garrido et al., 2014] 在 e-learning 适应性路径研究里就以先修图为核心数据结构。
  • Process Reward Models 的关键发现：[Lightman et al., 2023] ("Let's Verify Step by Step") 与 [Wang et al., 2024] (Math-Shepherd) 都指出"step-level 信号"比"trajectory-level 信号"更能驱动逐步推理。我们的 PhaseCoverage 是 step-level 检测的 lite 版本——本质上是启发式 process reward。
  • 关键词检测 ≠ 弱信号：[Dinucu-Jianu et al., 2025] 在教学法 RL 工作中显示，简单 lexical signal + composition 在 tutoring 任务上接近 LLM judge 的训练效果，但稳定性更高、不易 hack。
  • 抛弃 LLM judge 的合理性：[Zheng et al., 2023] 系统性记录了 LLM judge 的多种 bias（position bias、verbosity bias、family bias）；[Gao et al., 2023] 指出 reward model overoptimization 的根因是 judge 的非线性失效。规则化检测从根上消除这两类风险。

5. Reward Composition Argument

为什么"4 个 lite reward 组合" > "1 个完美 reward"：

正交化分工

Reward主要监督面主要失效模式被谁救场
RhardSchema / Cycle / Tool 合法性无（boolean，不可 hack）—
Rstruct结构相似度、DAG、依赖完整度boilerplate 模板能让 agent/subtask 集合层面匹配 goldRpers,lite（boilerplate 的 step instruction 短）；Rped,lite（boilerplate 缺 phase signature）
Rpers,lite步骤级 learner-grounded 改写200+ 字符的填充式 bridge 短语 → 形似但内容空Rstruct（填充式指令与 gold 文本差异巨大，fingerprint 跌）
Rped,lite教学阶段完整 + 顺序关键词 stuffing 能短期触发 phase 检测Rstruct（关键词堆砌偏离 gold 结构）+ 长度过滤继承自 Rpers,lite

实证：单 reward 不达标的 attack 在组合后被压低

在 v3 数据集 10 个 record 上，构造 3 种 attack：（1）keyword stuffing 进 backstory，（2）"You are a X developer" 每步重复，（3）200+ 字符的 bridge 短语填充。

ScenarioRpers,liteRtotal (composed)
Gold plan0.580.70
Attack 1 — backstory keyword stuffing0.000.55
Attack 2 — "You are a X developer" boilerplate0.000.55
Attack 3 — bridge-phrase stuffing0.000.55

Gap = +0.15 稳定且方向正确。没有任何单个 reward 完美判别，但组合后 gold 始终最高。

理论支撑

  • [Pan et al., 2022] 形式化论证"多 reward 的分布层耦合"如何对抗 misspecification。
  • [Skalse et al., 2022]：reward hacking 是单 reward 系统的固有问题；组合 reward 的 hacking surface 是各 reward hacking surface 的交集，难度成倍增加。
  • [Huang et al., 2025] MoGRPO 直接采用此设计思想：硬约束 + 多软偏好叠加。

核心选择

未实现的论文设计（明确列出）

  1. Segment-wise credit routing（论文 §4.2 Level 2，[Yu et al., 2025] webagent-r1 / [Li et al., 2025] RLTR 借鉴的 loss-mask 思路）。v1 用 scalar reward + 标准 advantage。代码留 subclass 草图 (rewards.py 末尾)。
  2. Adaptive β（当 KL 越过 budget 时自动放大）。v1 用固定 β=0.04。
  3. 5 条反作弊护栏（judge 随机化 / 对抗 trajectory 注入 / proxy-vs-gold 监控 / loss mask / adaptive β）。v1 通过 reward composition 实现了"软"反作弊；硬护栏作为升级路径保留。

理论支撑（GRPO + KL-to-SFT）

  • [Shao et al., 2024] DeepSeekMath：GRPO 的原始论文，确立组内相对优势 + KL 锚定的设计。
  • [DeepSeek-AI, 2025] R1：在 reasoning 任务上验证 GRPO + 验证奖励的有效性。
  • [Agarwal et al., 2024] on-policy distillation：从理论侧证明"KL 钉死预训练 / SFT"能把 exposure-bias 误差从 O(T²) 降到 O(T)。
  • [Zeng et al., 2025] SimpleRL-Zoo：在不同 backbone × 不同 reward 下系统对比 GRPO 训练稳定性，确认 KL 锚到固定 reference 是关键稳健化操作。
